## Under the hood
Что происходит с момента создания корутины до ее выполнения в потоке? Когда вы создаете корутину с помощью стандартных конструкторов корутин, вы можете указать, на каком [CoroutineDispatcher](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-core/common/src/CoroutineDispatcher.kt) ее запускать; если нет, то используется `Dispatchers.Default`.

**Диспетчер CoroutineDispatcher отвечает за диспетчеризацию выполнения coroutine в потоке**. Под капотом, когда используется `CoroutineDispatcher`, он перехватывает корутину с помощью метода [interceptContinuation](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-core/common/src/CoroutineDispatcher.kt#L99), который **заворачивает** `Continuation` в [DispatchedContinuation](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-core/common/src/internal/DispatchedContinuation.kt). Это возможно благодаря тому, что `CoroutineDispatcher` реализует интерфейс [ContinuationInterceptor](https://github.com/Kotlin/kotlinx.coroutines/blob/master/stdlib-stubs/src/ContinuationInterceptor.kt).

Если продолжение должно быть выполнено в другом диспетчере, метод `DispatchedContinuation` [resumeWith](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-core/common/src/internal/DispatchedContinuation.kt#L178) отвечает за отправку корутины в соответствующий диспетчер!

Более того,  `DispatchedContinuation` наследует абстрактный [DispatchedTask](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-core/common/src/internal/DispatchedTask.kt), который в реализации языка программирования Java представляет собой тип, реализующий интерфейс `Runnable`. Таким образом, `DispatchedContinuation` может выполняться в потоке! Насколько это круто? Если указать `CoroutineDispatcher`, то корутина преобразуется в `DispatchedTask`, которая отправляется на выполнение в поток как `Runnable`!

А теперь... Как вызывается метод `dispatch`, когда вы создаете coroutine? Когда вы создаете корутину с помощью стандартных конструкторов корутин, вы можете указать, как она начнется, с помощью параметра `start` типа [CoroutineStart](https://kotlin.github.io/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/-coroutine-start/index.html). Например, вы можете настроить его запуск только тогда, когда это необходимо, с помощью параметра `CoroutineStart.LAZY`. По умолчанию используется `CoroutineStart.DEFAULT`, который планирует выполнение coroutine в соответствии с его `CoroutineDispatcher`. Бинго!
## Dispatchers and thread pools
Вы можете выполнять корутины в любом из пулов потоков вашего приложения, преобразовав их в `CoroutineDispatcher` с помощью функции расширения [Executor.asCoroutineDispatcher()](https://kotlin.github.io/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/java.util.concurrent.-executor/as-coroutine-dispatcher.html). В качестве альтернативы можно использовать стандартные [Диспетчеры](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-core/jvm/src/Dispatchers.kt), которые поставляются в библиотеке coroutines.

Вы можете увидеть, как инициализируется `Dispatchers.Default` в этом методе [createDefaultDispatcher](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-core/jvm/src/CoroutineContext.kt#L22). По умолчанию используется [DefaultScheduler](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-core/jvm/src/scheduling/Dispatcher.kt). Если вы посмотрите реализацию [Dispatchers.IO](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-core/jvm/src/Dispatchers.kt#L118), она также использует `DefaultScheduler` и позволяет создавать не менее 64 потоков по требованию. `Dispatchers.Default` и `Dispatchers.IO` неявно связаны друг с другом, поскольку используют один и тот же пул потоков, что подводит меня к следующей теме. ### Производительность потоков и withContext
CoroutineScheduler, который является пулом потоков, используемым в реализации языка программирования Java по умолчанию, распределяет диспетчеризированные корутины по рабочим потокам наиболее эффективным образом. Поскольку Dispatchers.Default и Dispatchers.IO используют один и тот же пул потоков, переключение между ними оптимизировано таким образом, чтобы по возможности избежать переключения потоков. Библиотека coroutines может оптимизировать эти вызовы, оставаться в том же диспетчере и потоке и следовать по быстрому пути.

Поскольку Dispatchers.Main обычно является другим потоком в приложениях UI, переключение между Dispatchers.Default и Dispatchers.Main в coroutines не влечет за собой больших затрат производительности, поскольку coroutine просто приостанавливается (т. е. прекращает выполнение в одном потоке) и планируется к выполнению в другом потоке.
