- **Корутина** - экземпляр приостанавливаемого вычисления. Концептуально она похожа на поток в том смысле, что для ее выполнения требуется блок кода, и имеет аналогичный жизненный цикл - она создается и запускается, но не привязана к какому-либо конкретному потоку. Он может приостановить выполнение в одном потоке и возобновить его в другом. Более того, подобно future или promise, он может завершаться с некоторым результатом (который является либо значением, либо исключением) 

Завершения работы корутины, как и демон-потоков, JVM не ждет, поэтому, если у программы нет активных потоков, то она завершится, и работа, совершаемая в корутинах, пропадет

- **suspend-функция** - функция, помеченная модификатором `suspend`. Она может приостановить выполнение кода, не блокируя текущий поток выполнения, вызывая другие приостанавливающие функции. Приостанавливающая функция не может быть вызвана из обычного кода, а только из других приостанавливающих функций и suspend-лямбд (см. ниже). Стандартная библиотека предоставляет примитивные suspend-функции, которые используются для определения всех остальных.

- **suspend-лямбда** - блок кода, который выполняется в корутине. Он выглядит, как обычное [лямбда-выражение](https://kotlinlang.org/docs/reference/lambdas.html), но его тип помечен модификатором `suspend` modifier. Как обычное лямбда-выражение - это краткая синтаксическая форма для анонимной локальной функции, так и приостанавливающая лямбда - это краткая синтаксическая форма для анонимной приостанавливающей функции. Она может *приостанавливать* выполнение кода без блокирования текущего потока выполнения, вызывая приостанавливающие функции.

Обычные лямбды могут вызывать suspend-функции везде, где разрешен [нелокальный](https://kotlinlang.org/docs/reference/returns.html) возврат из лямбды.То есть вызовы приостанавливающих функций внутри встроенных лямбд, таких как блок apply{}, разрешены, но не в `noinline` и не в `crossinline` внутренних лямбда-выражениях. Приостановка рассматривается как особый вид нелокальной передачи управления

- **Тип suspend-функции** - это тип функции для приостановки функций и лямбд. Он похож на обычный тип функции, но с модификатором suspend. Например, `suspend () -> Int` - это тип приостанавливающей функции без аргументов, которая возвращает `Int`. Приостанавливающая функция, объявленная как `suspend fun foo(): Int`, соответствует этому типу функций.
- **Сoroutine builder** — функция, которая принимает в качестве аргумента некоторую suspend-лямбду, создает корутину и, опционально, предоставляет доступ к ее результату в той или иной форме. Стандартная библиотека предоставляет примитивные билдеры, которые используются для определения всех остальных билдеров
- **Точка приостановки** - это точка в процессе выполнения корутины, в которой выполнение корутины может быть приостановлено. Синтаксически точка приостановки - это вызов приостанавливающей функции, но фактическая приостановка происходит, когда приостанавливающая функция вызывает стандартный библиотечный примитив для приостановки выполнения.
- **Продолжение** — это состояние приостановленной корутины в точке приостановки. Концептуально представляет собой то, что осталось выполнить после точки приостановки
#### Интерфейс Continuation 
Это просто параметризованный колбек:
```Kotlin
interface Continuation<in T> {
   val context: CoroutineContext
   fun resumeWith(result: Result<T>)
}
```
```Kotlin
fun <T> Continuation<T>.resume(value: T)
fun <T> Continuation<T>.resumeWithException(exception: Throwable)
```
Из этих блоков, вызываемых внутренними билдерами, построены основные suspend-функции библиотеки. `resumeWith(result)` сообщает об успешном или неудачном завершении корутины
```Kotlin
suspend fun <T> suspendCoroutine(block: (Continuation<T>) -> Unit): T

fun <T> (suspend () -> T).startCoroutine(completion: Continuation<T>)

fun <R, T> (suspend R.() -> T).startCoroutine(  
    receiver: R,  
    completion: Continuation<T>)
```
`suspendCoroutine`, когда вызывается, фиксирует состояние выполнения корутины в экземпляре `Continuation` и передает это его в `block` в качестве аргумента. Чтобы возобновить выполнение программы, блок вызывает `continuation.resumeWith()` (либо напрямую, либо используя расширения `continuation.resume()` или `continuation.resumeWithException())` в этом потоке или в каком-то другом потоке в некоторое более позднее время. Фактическая приостановка работы корутины происходит, когда блок `suspendCoroutine` возвращается без вызова `resumeWith`. Если продолжение было возобновлено до возврата из этого блока, то корутина не считается приостановленной и продолжает выполняться. 

Возобновление одного и того же `Continuation` вызовет `IllegalStateException`, поэтому одно и то же состояние по корутине не гоняется.

`startCoroutine` создает корутину и сразу начинает выполнять в текущем потоке, до первой точки останова, (потом возвращает управление) без получателя (или с ним) и с типом результата T. При каждом вызове эта функция создает и запускает новый, свежий экземпляр приостанавливаемого вычисления.  Продолжение завершения вызывается, когда корутина завершается с результатом или исключением. 

Точка приостановки - это вызов некоторой приостанавливающей функции в теле корутины, и только код соответствующей приостанавливающей функции определяет, когда и как возобновляется выполнение корутины.

Завершение работы корутины вызывает ее продолжение завершения. Его функция `resumeWith` вызывается, когда корутины завершается с успехом или неудачей. Поскольку `launch` делает корутину "fire-and-forget", она определена для приостановки функций с типом возврата Unit и фактически игнорирует этот результат в своей функции `resume`. Если выполнение корутины завершается с исключением, то для сообщения об этом используется обработчик непойманных исключений текущего потока. 
#### Контекст корутины
Контекст корутины - это неизменяемое множество определяемых пользователем объектов, которые могут быть прикреплены к корутине. Он может включать объекты, отвечающие за политику потоков, протоколирование, безопасность и транзакционные аспекты выполнения программы, идентификацию и имя программы и т. д. 

Можно считать корутину это облегченным потоком. В этом случае контекст корутины подобен коллекции локальных переменных потока. Разница в том, что локальные переменные потока изменяемы, а контекст корутины неизменяем, что не является серьезным ограничением для корутин, поскольку они настолько легки, что можно запустить новую корутину, когда контекст нужно изменить.

Концептуально контекст корутины - это индексированный набор элементов, где каждый элемент имеет уникальный ключ. Это нечто среднее между множеством и таблицей. Его элементы имеют ключи, как в карте, но ключи непосредственно связаны с элементами, как в множестве. В стандартной библиотеке определен минимальный интерфейс для `CoroutineContext` (в пакете `kotlin.coroutines`)
```kotlin
interface CoroutineContext {
    operator fun <E : Element> get(key: Key<E>): E?
    fun <R> fold(initial: R, operation: (R, Element) -> R): R
    operator fun plus(context: CoroutineContext): CoroutineContext
    fun minusKey(key: Key<*>): CoroutineContext

    interface Element : CoroutineContext {
        val key: Key<*>
    }

    interface Key<E : Element>
}
```
`Element` контекста корутины сам является контекстом. Это контекст-одиночка, содержащий только этот элемент. Это позволяет создавать составные контексты, беря библиотечные определения элементов контекста корутины и соединяя их с помощью `+`. Например, если одна библиотека определяет элемент `auth` с информацией об авторизации пользователя, а другая - объект `threadPool` с информацией о контексте выполнения, то вы можете использовать конструктор корутин `launch{}` с объединенным контекстом, используя вызов `launch(auth + threadPool) {...}`
#### Перехватчик продолжения
То, что в имплементации `Continuation` осуществляет передачу состояния выполнения лямбды в корутине.
```kotlin
interface ContinuationInterceptor : CoroutineContext.Element {
    companion object Key : CoroutineContext.Key<ContinuationInterceptor>
    fun <T> interceptContinuation(continuation: Continuation<T>): Continuation<T>
    fun releaseInterceptedContinuation(continuation: Continuation<*>)
}
```
Перехватчик продолжения имеет возможность перехватить и обернуть продолжение от его возобновления до последующих точек приостановки. Начальный код корутины рассматривается как возобновление ее начального продолжения. Стандартная библиотека предоставляет интерфейс [ContinuationInterceptor](https://kotlinlang.org/api/latest/jvm/stdlib/kotlin.coroutines/-continuation-interceptor/) (в пакете `kotlin.coroutines`). Всякий раз, когда корутина приостанавливается, фреймворк использует следующую строку кода для обертывания фактического продолжения для последующего возобновления
````kotlin
val intercepted = continuation.context[ContinuationInterceptor]?.interceptContinuation(continuation) ?: continuation
````
Фреймворк кэширует полученное перехваченное продолжение для каждого фактического экземпляра продолжения и вызывает `releaseInterceptedContinuation(intercepted)`, когда оно больше не нужно.

Продолжение перехватывается только тогда, когда фактическая приостановка происходит во время выполнения корутины, то есть когда блок `suspendCoroutine` возвращается без вызова `resume`
#### Генераторы 
Билдер `sequence` умеет делать ленивые коллекции, потому что там метод `yield` под капотом. Эти функции формируют *ограниченную область видимости приостанова*, потому что работают синхронно и сами хранят, захватывают и продолжают `continuation`
#### Программирование с передачей продолжения
[Началось с Лиспа](https://ru.wikipedia.org/wiki/%D0%9F%D1%80%D0%BE%D0%B4%D0%BE%D0%BB%D0%B6%D0%B5%D0%BD%D0%B8%D0%B5_(%D0%B8%D0%BD%D1%84%D0%BE%D1%80%D0%BC%D0%B0%D1%82%D0%B8%D0%BA%D0%B0). Это когда состояние программы в определенный момент передается как параметр.

Каждая suspend-функция и лямбда имеет доп. параметр `Continuation`, который неявно передается ей при вызове. Там состояние и хранится.

Компилятор создает только один класс для приостанавливающей лямбды, которая может иметь произвольное количество точек приостановки в своем теле.

Основная идея: приостанавливающая функция компилируется в конечный автомат, состояния которого соответствуют точкам приостановки. Пример: возьмем приостанавливающий блок с двумя точками приостановки:
```Kotlin
val a = a()
val y = foo(a).await() // suspension point #1
b()
val z = bar(a, y).await() // suspension point #2
c(z)
```
У этого блока кода есть три состояния:

- начальное (до любой точки приостановки)
- после первой точки приостановки
- после второй точки приостановки

Каждое состояние является точкой входа в одно из продолжений этого блока (начальное продолжение продолжается с самой первой строки).

Код компилируется в анонимный класс, который содержит метод, реализующий конечный автомат, поле, хранящее текущее состояние машины состояний, и поля для локальных переменных корутины, которые разделяются между состояниями (также могут быть поля для закрытия корутины, но в данном случае они пусты). Вот псевдо-Java-код для приведенного выше блока, использующего стиль передачи продолжения для вызова приостанавливающих функций `await`:
```Kotlin
class <anonymous_for_state_machine> extends SuspendLambda<...> {
    // The current state of the state machine
    int label = 0
    
    // local variables of the coroutine
    A a = null
    Y y = null
    
    void resumeWith(Object result) {
        if (label == 0) goto L0
        if (label == 1) goto L1
        if (label == 2) goto L2
        else throw IllegalStateException()
        
      L0:
        // result is expected to be `null` at this invocation
        a = a()
        label = 1
        result = foo(a).await(this) // 'this' is passed as a continuation 
        if (result == COROUTINE_SUSPENDED) return // return if await had suspended execution
      L1:
        // external code has resumed this coroutine passing the result of .await() 
        y = (Y) result
        b()
        label = 2
        result = bar(a, y).await(this) // 'this' is passed as a continuation
        if (result == COROUTINE_SUSPENDED) return // return if await had suspended execution
      L2:
        // external code has resumed this coroutine passing the result of .await()
        Z z = (Z) result
        c(z)
        label = -1 // No more steps are allowed
        return
    }          
}
```
#### Компиляция suspend-функций
Компилируемый код SF зависит от того, как и когда она вызывает другие SF. В простейшем случае вызывает только в хвостовых позициях, делая к ним хвостовые вызовы. Это типичный случай для SF, реализующих низкоуровневые примитивы синхронизации или оборачивающие колбеки. Эти функции вызывают другую SF, например suspendCoroutine, в хвостовой позиции. Они компилируются так же, как и обычные, с тем лишь исключением, что неявный параметр `Continuation`, полученный ими в результате преобразования CPS, передается следующей SF в хвостовом вызове.

В случае, когда приостанавливающие вызовы появляются в нехвостовых позициях, компилятор создает конечный автомат для соответствующей функции. Его экземпляр создается при вызове suspend-функции и удаляется по ее завершении.

Этот конечный автомат, в свою очередь, служит продолжением завершения для вызова других SF в нехвостовых позициях. Этот экземпляр обновляется и повторно используется, когда функция выполняет несколько обращений к другим SF. Сравните это с другими стилями асинхронного программирования, где каждый последующий шаг асинхронной обработки обычно реализуется с помощью отдельного, только что выделенного объекта замыкания.
#### Внутренности корутин
Собственно реализация функции приостановки корутин `suspendCoroutine` в стандартной библиотеке написана на языке Kotlin, и ее исходный код доступен в составе пакета исходных текстов стандартной библиотеки. Чтобы обеспечить безопасное и беспроблемное использование корутин, она оборачивает действительное продолжение конечного автомата в дополнительный объект при каждой приостановке. Это совершенно нормально для действительно асинхронных случаев использования, таких как асинхронные вычисления и future, поскольку накладные на выполнение соответствующих асинхронных примитивов намного превышают затраты на дополнительный выделенный объект. Однако для случаев использования генераторов эти дополнительные затраты непомерны, поэтому пакеты intrinsics предоставляют примитивы для чувствительного к производительности низкоуровневого кода.

[`suspendCoroutineUninterceptedOrReturn`](http://kotlinlang.org/api/latest/jvm/stdlib/kotlin.coroutines.intrinsics/suspend-coroutine-unintercepted-or-return.html) дает прямой доступ к передаче `Continuation` и раскрывает неперехваченную ссылку на него. Последнее означает, что вызов `Continuation.resumeWith` не проходит через `ContinuationInterceptor`. Это может быть использовано при написании синхронных корутин с ограничением приостанова, которые не могут иметь установленного перехватчика (так как их контекст всегда пуст), или когда известно, что текущий выполняющийся поток уже находится в нужном контексте. В противном случае перехваченное продолжение должно быть получено с помощью функции расширения intercepted (из пакета kotlin.coroutines.intrinsics):
```Kotlin
fun <T> Continuation<T>.intercepted(): Continuation<T>
```
и `Continuation.resumeWith` должен быть вызван для полученного перехваченного продолжения.

Теперь блок, переданный функции `suspendCoroutineUninterceptedOrReturn`, может вернуть маркер `COROUTINE_SUSPENDED`, если корутина действительно приостановилась (в этом случае `Continuation.resumeWith `должна быть вызвана ровно один раз позже), или вернуть значение результата `T`, или выбросить исключение (в обоих последних случаях `Continuation.resumeWith` никогда не должна быть вызвана).

Несоблюдение этого соглашения при использовании `suspendCoroutineUninterceptedOrReturn` приводит к трудноотслеживаемым ошибкам, которые не поддаются попыткам найти и воспроизвести их с помощью тестов. Это соглашение обычно легко соблюдается для `buildSequence/yield`-подобных корутин, но попытки написать асинхронные await-подобные приостанавливающие функции поверх `suspendCoroutineUninterceptedOrReturn` **не приветствуются**, так как их очень сложно правильно реализовать без помощи `suspendCoroutine`.

Существуют также функции `createCoroutineUnintercepted` (из пакета `kotlin.coroutines.intrinsics`) со следующими сигнатурами:
```kotlin
fun <T> (suspend () -> T).createCoroutineUnintercepted(completion: Continuation<T>): Continuation<Unit>
fun <R, T> (suspend R.() -> T).createCoroutineUnintercepted(receiver: R, completion: Continuation<T>): Continuation<Unit>
```
Они работают аналогично `createCoroutine`, но возвращают неперехваченную ссылку на начальное продолжение. Аналогично `suspendCoroutineUninterceptedOrReturn`, он может быть в синхронных корутинах для повышения производительности.
Например, ниже показан оптимизированный вариант построения `sequence{}` через `createCoroutineUnintercepted`:
```kotlin
fun <T> sequence(block: suspend SequenceScope<T>.() -> Unit): Sequence<T> = Sequence {
    SequenceCoroutine<T>().apply {
        nextStep = block.createCoroutineUnintercepted(receiver = this, completion = this)
    }
}
```
Оптимизированная версия `yield` через `suspendCoroutineUninterceptedOrReturn` показана ниже. Обратите внимание, что поскольку `yield` всегда приостанавливает выполнение, соответствующий блок всегда возвращает `COROUTINE_SUSPENDED`.
```Kotlin
// Реализация генератора
override suspend fun yield(value: T) {
    setNext(value)
    return suspendCoroutineUninterceptedOrReturn { cont ->
        nextStep = cont
        COROUTINE_SUSPENDED
    }
}
```
Два дополнительных интринсика предоставляют низкоуровневую версию `startCoroutine` и называются `startCoroutineUninterceptedOrReturn`:
```Kotlin
fun <T> (suspend () -> T).startCoroutineUninterceptedOrReturn(completion: Continuation<T>): Any?
fun <R, T> (suspend R.() -> T).startCoroutineUninterceptedOrReturn(receiver: R, completion: Continuation<T>): Any?
```
Они отличаются от `startCoroutine` в двух аспектах. Во-первых, `ContinuationInterceptor` не используется автоматически при запуске корутины, поэтому вызывающая сторона должна обеспечить правильный контекст выполнения, если это необходимо. Во-вторых, если корутина не приостанавливается, но возвращает значение или выбрасывает исключение, то вызов `startCoroutineUninterceptedOrReturn` возвращает это значение или выбрасывает это исключение. Если корутина приостанавливается, то возвращается `COROUTINE_SUSPENDED`.
Основное использование `startCoroutineUninterceptedOrReturn` - это сочетание его с `suspendCoroutineUninterceptedOrReturn` для продолжения выполнения приостановленной корутины в том же контексте, но с другим блоком кода:
```Kotlin
suspend fun doSomething() = suspendCoroutineUninterceptedOrReturn { cont ->
    // выясняем или создаем блок кода, который нужно запустить
    startCoroutineUninterceptedOrReturn(completion = block) // возвращаем результат в suspendCoroutineUninterceptedOrReturn 
}
```
