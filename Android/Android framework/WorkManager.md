WorkManager - это библиотека для Android, которая запускает отложенную фоновую работу, когда ее выполнены ее требования.

WorkManager предназначен для задач, которые требуют гарантии, что система запустит их даже при выходе из приложения.

| Feature            | android-job | WorkManager |
| ------------------ | ----------- | ----------- |
| Exact jobs         | Yes         | No          |
| Transient jobs     | Yes         | No          |
| Daily jobs         | Yes         | No          |
| Custom Logger      | Yes         | No          |
| Observe job status | No          | Yes         |
| Chained jobs       | No          | Yes         |
| Work sequences     | No          | Yes         |
### Как заставляет работать фоном
Пока приложение на переднем плане, мы можем запускать любые сервисы для фоновых задач. Но когда приложение переводится в фон, оно получает от системы еще несколько минут на выполнение задач, после чего по воле Android отключается. У foreground сервиса меньше шансов на уничтожение, но система все равно попытается за счет фонового процесса освободить ресурсы для основного на этот момент приложения. В этой ситуации мы не можем надеяться на то, что наши задачи будут выполнены – и тем более возобновлены позже. Выполнение длительных задач в этой ситуации становится практически невозможным.

WorkManager как раз решает эту проблему, помогая запустить длительную работу и гарантируя устойчивость ее выполнения.

Чтобы сделать это возможным, WorkManager связывает жизненный цикл процесса, в котором выполняется работа, с жизненным циклом foreground service. WorkManager по-прежнему осведомлен об этой работе, но foreground service полностью владеет жизненным циклом выполнения.

Так как foreground service требует показ уведомления пользователю, разработчики добавили API для WorkManager. Также WorkManager предоставляет API-интерфейсы, чтобы пользователь мог остановить выполнение работы прямо из уведомления.

Помимо более простого и последовательного API, WorkManager обладает рядом других ключевых преимуществ:
### Ограничения работы
Декларативно определяйте оптимальные условия для выполнения работы с помощью ограничений работы. Например, запускать только тогда, когда устройство находится в сети без подзарядки, когда устройство простаивает или когда у него достаточно заряда батареи.
### Надежное планирование
WorkManager позволяет планировать работу на однократное или многократное выполнение с помощью гибких окон планирования. Работы можно помечать и присваивать им имена, что позволяет планировать уникальные, заменяемые работы, а также контролировать или отменять группы работ вместе.

Запланированные работы хранятся во внутренней базе данных SQLite, и WorkManager заботится о том, чтобы эти работы сохранялись и перепланировались при перезагрузке устройства.

Кроме того, WorkManager придерживается энергосберегающих функций и лучших практик, таких как режим Doze, так что вам не придется об этом беспокоиться.
### Ускоренная работа
С помощью WorkManager можно запланировать немедленную работу для выполнения в фоновом режиме. Ускоренную работу следует использовать для важных для пользователя задач, которые выполняются в течение нескольких минут.
### Гибкая политика повторных попыток
Иногда работа не удается. WorkManager предлагает гибкую политику повторных попыток, включая настраиваемую экспоненциальную политику обратного хода.
### Создание цепочек работ
Для выполнения сложной связанной работы объедините отдельные задания в цепочку с помощью интуитивно понятного интерфейса, который позволяет контролировать, какие части выполняются последовательно, а какие - параллельно.
### За кулисами - как проходит работа
По умолчанию WorkManager будет:
- Запускать вашу работу из главного потока (это предполагает, что вы расширяете класс Worker).
- Гарантировать выполнение вашей работы (он не забудет запустить ее, даже если вы перезагрузите устройство или выйдете из приложения).
- Выполнение в соответствии с лучшими практиками для пользовательского уровня API (как описано в предыдущей статье).

За кулисами WorkManager включает в себя следующие части:

Внутренний `TaskExecutor`: Однопоточный исполнитель, который обрабатывает все запросы на выполнение работы.
- База данных `WorkManager`: Локальная база данных, которая отслеживает всю информацию и статусы всех ваших работ. Сюда входят такие сведения, как текущее состояние работы, входы и выходы работы, а также любые ограничения на работу. Эта база данных позволяет `WorkManager` гарантировать завершение работы - если устройство пользователя перезагружается и работа прерывается, все детали работы могут быть извлечены из базы данных, и работа может быть перезапущена, когда устройство снова загрузится.
- WorkerFactory**: Фабрика по умолчанию, которая создает экземпляры ваших `Workers`.
- Default Executor**: Исполнитель по умолчанию, который запускает вашу работу, если вы не укажете иное. Это гарантирует, что по умолчанию ваша работа будет выполняться синхронно и вне основного потока.
** Эти части могут быть переопределены, чтобы иметь различное поведение.
![[workmanager_internally.webp]]
Когда вы регистрируете свой WorkRequest:
1. Internal TaskExecutor немедленно сохраняет информацию о WorkRequest в базе данных WorkManager.
2. Позже, когда ограничения для WorkRequest будут выполнены (что может произойти немедленно), внутренний TaskExecutor сообщает WorkerFactory о создании Worker.
3. Затем стандартный исполнитель вызывает метод doWork() вашего работника из главного потока.
Таким образом, ваша работа по умолчанию гарантированно будет выполнена и запущена в главном потоке.

Или вы можете указать, как именно будет выполняться работа, используя ListenableWorker. Worker - это фактически реализация ListenableWorker, которая по умолчанию запускает вашу работу на исполнителе по умолчанию и, таким образом, синхронно. Поэтому, если вы хотите получить полный контроль над стратегией потоков в вашей работе или запустить работу асинхронно, вы можете унаследоваться от ListenableWorker (подробности об этом будут рассмотрены в следующем посте).

Тот факт, что WorkManager берет на себя труд сохранять всю информацию о вашей работе в базе данных, делает его идеальным для задач, выполнение которых должно быть гарантировано. Это также делает WorkManager излишним для задач, которым не нужна такая гарантия и которые просто должны выполняться в фоновом потоке.
