### Плагин компилятора Kotlin
Компилятор Compose на самом деле является плагином компилятора Kotlin. Это дает библиотеке возможность встраивать свою работу во время компиляции в фазы компиляции Kotlin, получая доступ к более релевантной информации о форме кода и ускоряя общий процесс. В то время как kapt должен запускаться до компиляции, плагин компилятора полностью встраивается в процесс компиляции

Будучи плагином компилятора Kotlin, он также дает возможность напрямую интегрировать проверки Compose с проверками idea, так что разработчик может получать обратную связь в процессе кодирования без необходимости собирать проект. Это происходит потому, что плагины компилятора имеют возможность сообщать о предупреждениях и ошибках на этапе фронтенда компилятора, а этот механизм уже интегрирован с idea. Улучшение цикла обратной связи - это конечная выгода от выполнения статического анализа на этапе фронтенда компилятора Kotlin.

Еще одно большое преимущество плагинов для компилятора Kotlin заключается в том, что они могут по своему усмотрению корректировать исходники (а не только добавлять новый код), настраивая выходное промежуточное представление для этих элементов, прежде чем они будут опущен до более атомарных терминов, которые затем могут быть переведены в примитивы, поддерживаемые целевыми платформами - напомним, что Kotlin мультиплатформенен. Мы немного подробнее остановимся на этом в данной главе, но это даст компилятору Compose возможность преобразовывать функции `Composable` в поддерживаемые средой выполнения. 
### Аннотации Compose
Первое, что происходит при запуске программы с помощью JC, - исходный код должен быть обработан библиотекой. Компилятор Compose ищет в нем все вхождения аннотации `@Composable`. При этом Compose также предоставляет другие дополнительные аннотации, которые позволяют использовать дополнительные проверки и различные оптимизации во время выполнения или "ярлыки" при некоторых определенных обстоятельствах. Все доступные аннотации предоставляются библиотекой времени выполнения Compose.

Все аннотации Jetpack Compose предоставляются средой выполнения Compose, поскольку и компилятор, и среда выполнения являются модулями, использующими эти аннотации.
##### @Composable
Эффективно изменяет тип аннотируемой функции, делая ее перезапускаемой - так что она может выполняться много раз - и представляет собой отображение данных на узел, который выдается в дерево после выполнения. Этот узел может быть узлом пользовательского интерфейса или узлом любой другой природы, в зависимости от библиотеки, которую мы используем для работы с Compose. Среда выполнения JC работает с общими типами узлов, не привязанными к какому-либо конкретному сценарию использования или семантике. 
##### @ExperimentalComposeApi
Для пометки API Compose, которые все еще открыты для изменений, прежде чем станут стабильными. Это означает, что при использовании этих API в нашей кодовой базе будут появляться предупреждения, и нам нужно будет явным образом отказаться от их использования. Фронтенд-проверка компилятора позаботится о том, чтобы эти предупреждения появлялись как можно раньше.
##### @ComposeCompilerApi
Эта аннотация используется самим Compose, чтобы отметить некоторые его части, которые должны быть доступны только компилятору, поэтому при попытке использовать их вне Compose вы получите ошибки компиляции
##### @InternalComposeApi
Некоторые API помечены в Compose как внутренние, поскольку ожидается, что они будут меняться внутри модуля, даже если публичная часть API останется неизменной и застынет на пути к стабильному релизу. Эта аннотация имеет более широкую область применения, чем ключевое слово `internal`, так как позволяет использовать его в разных модулях, а это концепция, которую Kotlin не поддерживает.
##### @DisallowComposableCalls
Используется для предотвращения вызовов `Composable` внутри функции. Это может быть полезно для встроенных лямбда-параметров CF, которые не могут безопасно содержать вызовы `Composable`.
Пример этого можно найти в `ComposeNode`, части Compose UI. Этот `Composable` эмитирует узел UI в композицию. При его вызове необходимо передать лямбды для инициализации или обновления созданного Composable.
```Kotlin
@Composable inline fun <T : Any, reified E : Applier<*>> ComposeNode(
	noinline factory: () -> T,  
	update: @DisallowComposableCalls Updater<T>.() -> Unit
	) {
	///...
}
```
Лямбда `update`является `inline` и также помечена как `@DisallowComposableCalls`, поскольку рантайм будет вызывать эту лямбду `inline` при каждом выбросе, а значит, при каждой рекомпозиции. Учитывая, что новые СС в этот момент композиции не будут ни ожидаться, ни поддерживаться, рантайм позаботится о том, чтобы запретить их.

Под ведро эту аннотацию пользовать незачем, но она может стать более актуальной, если вы используете Jetpack Compose не для Compose UI, а для других целей. В этом случае вам, скорее всего, придется написать собственную клиентскую библиотеку для рантайма, и это потребует от вас соблюдения ограничений рантайма.
##### @ReadOnlyComposable
Тело этой CF никогда не будет записывать в `Composition`, а только читать оттуда. Это также должно оставаться верным для всех вложенных вызовов `Composable` в теле. Это позволяет рантайму избежать генерации кода, которая не понадобится, если `Composable` сможет выполнить это предположение.

Для любой `Composable`, которая пишет в композицию внутри себя, компилятор генерирует "группу", которая оборачивает его тело, поэтому во время выполнения выдается вся группа. Эти испускаемые группы предоставляют композиции необходимую информацию о CF, чтобы она знала, как очистить записанные данные позже, когда при рекомпозиции потребуется заменить их данными другой CF, или как перемещать эти данные, сохраняя идентичность `Composable`. Существуют различные типы групп, которые могут быть созданы: Например, перезапускаемые группы, перемещаемые группы... и т. д.

Чтобы понять, что такое "группа", представьте себе пару указателей в начале и конце определенного участка выделенного текста. У всех групп есть исходный позиционный ключ, который используется для хранения группы и, следовательно, разблокирует позиционную мемоизацию. Этот ключ также позволяет определить тождество между ветвями if и else в таких условных логиках, как:
```Kotlin
if (condition) {  
	Text("Hello")  
} else {  
	Text("World") 5}
```
Обе CF - `Text`, но они имеют разную идентичность, поскольку представляют разные идеи для вызывающей функции. Перемещаемые группы также имеют семантический ключ идентичности, поэтому они могут переупорядочиваться внутри своей родительской группы. 

Когда наш компонент не пишет в композицию, создание этих групп не приносит никакой пользы, поскольку его данные не будут заменены или перемещены. Эта аннотация помогает избежать этого.

Примерами CF, доступных только для чтения, в библиотеках Compose могут быть многие значения `CompositionLocal` по умолчанию или утилиты, которые делегируют эти значения, например, цвета материалов, типографика, функция `isSystemInDarkTheme()`, `LocalContext`, любые вызовы для получения ресурсов приложения любого типа - поскольку они полагаются на `LocalContext` - или `LocalConfiguration`. В общем, речь идет о вещах, которые устанавливаются только один раз при выполнении нашей программы, и ожидается, что они останутся неизменными и будут доступны для чтения из CF в дереве.
##### @NonRestartableComposable
Применяясь к функции или геттеру свойства, она, по сути, заставляет их быть неперезапускаемыми CF.

При добавлении этого свойства компилятор не генерирует требуемые шаблоны, необходимые для того, чтобы функция могла рекомпозироваться или быть пропущенной во время рекомпозиции. Пожалуйста, имейте в виду, что это нужно использовать очень осторожно, так как это может иметь смысл только для очень маленьких функций, которые, скорее всего, будут перекомпонованы (перезапущены) другой вызывающей их Composable-функцией, так как они могут содержать очень мало логики, поэтому для них нет особого смысла самоликвидироваться. Другими словами, их аннулирование/рекомпозиция будет в основном определяться их родительской/окружающей CF.
##### @StableMarker
Аннотирует другие аннотации, такие как `@Immutable` и `@Stable`. Это может показаться немного избыточным, но она предназначена для многократного использования, поэтому импликации, которые она имеет, также применяются ко всем аннотациям, аннотированным с ее помощью.
Маркер `@StableMarker` подразумевает следующие требования, связанные со стабильностью данных для конечного аннотированного типа:
- Результат вызова `equals` всегда будет одинаковым для двух одинаковых экземпляров.
- Композиция всегда получает уведомление об изменении публичного свойства аннотированного типа.
- Все публичные свойства аннотированного типа также стабильны.

Обратите внимание, что это обещания, которые мы даем компилятору, чтобы он при обработке наших исходников мог сделать некоторые предположения, но они не проверяются во время компиляции. **Это означает, что ответственность за эту аннотацию и другие, описанные ниже, лежит на разработчике, использующем их, и должна использоваться только тогда, когда вы точно уверены, что все их требования выполнены**
##### @Immutable
Эта аннотация применяется к классу как строгое обещание компилятору, что все общедоступные свойства и поля класса останутся неизменными после создания. Обратите внимание, что это более строгое обещание, чем ключевое слово `val` в языке, поскольку `val` гарантирует только то, что свойство не может быть переназначено через сеттер, но оно может указывать, например, на мутабельную структуру данных, делая наши данные мутабельными, даже если они имеют только свойства `val`. Это нарушит ожидания среды выполнения Compose. Другими словами, эта аннотация нужна Compose в основном потому, что язык Kotlin не предоставляет механизма (ключевого слова или чего-то подобного), гарантирующего, что некоторая структура данных является неизменяемой.

Основываясь на предположении, что значение, считываемое из типа, никогда не изменится после инициализации, среда выполнения может применить оптимизацию к функциям умной рекомпозиции и пропуска рекомпозиции.

Хорошим примером класса, который можно смело пометить как `@Immutable`, будет класс данных только с val-свойствами, где ни у одного из них нет пользовательских геттеров - которые иначе будут вычисляться при каждом вызове и потенциально возвращать каждый раз разные результаты, превращая его в нестабильный API для чтения - и все они являются либо примитивными типами, либо типами, также помеченными как `@Immutable`.
##### @Stable
Когда эта аннотация применяется к типу, это означает, что тип является изменяемым, и он будет иметь только те импликации, которые наследует `@StableMarker`.

Когда аннотация `@Stable` применяется к функции или свойству, она сообщает компилятору, что функция всегда будет возвращать один и тот же результат при одних и тех же входных данных (pure). Это возможно только в том случае, если параметры функции также `@Stable`, `@Immutable` или примитивы (считаются стабильными)

В документации есть хороший пример того, как это важно для среды выполнения: *Если все типы, передаваемые в качестве параметров в функцию Composable, помечены как стабильные, то значения параметров сравниваются на равенство на основе позиционной мемоизации и вызов пропускается, если все значения равны предыдущему вызову*.

Примером типа, который может быть помечен как `@Stable`, является объект, чьи публичные свойства не изменяются, но не могут считаться неизменяемыми. Например, он имеет приватное изменяемое состояние или использует делегирование свойств объекту `MutableState`, но в остальном является неизменяемым с точки зрения его использования извне.

Еще раз повторимся, что последствия этой аннотации используются компилятором и средой выполнения для того, чтобы сделать предположения о том, как будут развиваться (или не развиваться) наши данные, и при необходимости использовать короткие пути. И еще раз повторю, что эту аннотацию никогда не следует использовать, если вы не уверены в том, что ее последствия полностью оправданы. В противном случае мы предоставим компилятору неверную информацию, что легко приведет к ошибкам во время выполнения. Вот почему все эти аннотации рекомендуется использовать редко.

Интересно отметить, что даже если аннотации `@Immutable` и `@Stable` - это разные обещания с разным смыслом, компилятор Jetpack Compose сегодня воспринимает их одинаково: Чтобы включить и оптимизировать умную рекомпозицию и пропуск рекомпозиций. Тем не менее обе аннотации существуют для того, чтобы оставить дверь открытой для различных семантик, чтобы наложить различия, которые компилятор и среда выполнения могут захотеть использовать в будущем.
### Регистрация расширений компилятора
Первое, что делает плагин компилятора Compose, - регистрирует себя в конвейере компилятора Kotlin с помощью `ComponentRegistrar` - механизма, который компилятор Kotlin предоставляет для этого. `ComposeComponentRegistrar` регистрирует ряд расширений компилятора для различных целей. Эти расширения будут отвечать за облегчение работы с библиотекой и генерацию необходимого кода для среды выполнения.

Компилятор Compose также регистрирует несколько расширений в зависимости от включенных флагов компилятора. Разработчики, использующие JC, могут включить несколько специфических флагов компилятора, которые позволяют им включить такие возможности, как живые литералы, включение исходной информации в генерируемый код, чтобы Android Studio и другие инструменты могли проверить композицию, оптимизация для запоминающих функций, подавление проверок совместимости версий Kotlin и/или генерация ложных методов в IR-преобразовании.
### Статический анализ
В соответствии со стандартным поведением обычного плагина компилятора, первое, что происходит, - это линтинг. Статический анализ выполняется путем сканирования исходного кода в поисках любых библиотечных аннотаций, а затем выполнением несколько важных проверок, чтобы среда выполнения убедилась, что они используются правильно. Здесь соответствующие предупреждения или ошибки сообщаются через контекстную трассировку, к которой имеют доступ плагины компилятора. Это хорошо интегрируется с IDEA, так как она уже готова отобразить эти предупреждения или ошибки в строке, пока разработчик продолжает набирать текст. Как уже говорилось ранее, все эти проверки происходят на этапе фронтенда компилятора, что помогает Compose обеспечить максимально быстрый цикл обратной связи для разработчиков.
#### Версия компилятора Kotlin
Компилятор Compose требует очень специфическую версию Kotlin, поэтому он проверяет, совпадает ли используемая версия компилятора Kotlin с требуемой. Это первая проверка, так как в случае невыполнения работа сильно затруднится.

Есть возможность обойти эту проверку, используя аргумент компилятора `suppressKotlinVersionCompatibilityCheck`, но это на наш страх и риск, поскольку тогда мы получаем возможность запускать Compose с любой версией Kotlin, что легко может привести к важным несоответствиям. 
#### Статические проверки
Некоторые из зарегистрированных расширений представлены в виде статических проверок, которые помогут разработчикам при написании кода. Проверки вызовов, типов и объявлений зарегистрированы в качестве расширений Jetpack Compose. Они обеспечивают правильное использование библиотеки и, очевидно, имеют свое мнение о проблеме. которую хочет решить библиотека. Такие вещи, как требования к CF, проверяются здесь и сообщаются в случае нарушения.

Учитывая, что все зарегистрированные проверки работают на этапе фронтенда компилятора Kotlin, ожидается, что они будут очень быстрыми и не будут содержать очень требовательных к процессору операций.
#### Проверки вызовов
Компилятор Compose имеет статические проверки вызовов для проверки вызовов CF во многих различных контекстах, например, когда они выполняются в рамках `@DisallowComposableCalls` или `@ReadOnlyComposable`.
Проверка вызова - это расширение компилятора, используемое для выполнения статического анализа всех вызовов в нашей кодовой базе, поэтому оно предоставляет функцию `check`, которая рекурсивно вызывается для посещения всех элементов, которые считаются вызовами в наших источниках. Или, другими словами: Все узлы на дереве PSI. Это реализация паттерна Посетитель.

Некоторые из этих проверок требуют более широкого контекста, чем текущий элемент языка, который они посещают, поскольку им может потребоваться знать, например, откуда вызывается `Composable`. Это означает, что анализа одного узла PSI недостаточно. Для сбора такой информации необходимо записывать более мелкие кусочки информации из разных посещенных элементов, как навигационную цепочку, чтобы построить историю в целом и выполнить более сложные проверки при дальнейших проходах. Для этого компилятор может удобно записывать эту информацию в контекстную **трассировку**. Это позволяет расширить область применения проверок и искать вложенные лямбда-выражения, блоки `try / catch` и т.д.

Другие проверки более просты и требуют только информацию, доступную о текущем посещенном элементе, поэтому они выполняют свое действие и возвращаются. При каждом вызове проверки плагин сопоставляет тип элемента с текущим узлом, и в зависимости от этого просто выполняет проверку и возвращается - если все правильно, - сообщает об ошибке - если нужно, - записывает соответствующую информацию в контекстную трассировку или снова обращается к родителю этого узла, чтобы продолжить посещение других узлов и собрать больше информации. На этом пути выполняются различные проверки для разных аннотаций.

Компилятор Compose проверяет, что CF не вызываются из запрещенных мест, например, из блока `try/catch` (который не поддерживается), из функции, не аннотированной как `Composable`, или из лямбд, аннотированных с `@DisallowComposableCalls`. 

Для каждого вызова CF компилятор поднимается по дереву PSI, проверяя его вызывающих, вызывающих его вызывающих и так далее, чтобы убедиться, что все требования для этого вызова выполнены. При этом учитываются все сценарии, поскольку родителями могут быть лямбда-выражения, функции, свойства, получатели доступа к свойствам, блоки `try/catch`, классы, файлы и многое другое.

Также важно, чтобы эти проверки учитывали `inline` функции, поскольку вызов CF из встроенной лямбды должен быть возможен до тех пор, пока вызывающие ее лямбды также являются `Composable`. Компилятор проверяет, что все встроенные лямбды, вызывающие CF, также заключены в CF на каком-то уровне вверх по стеку вызовов.

Другая проверка, выполняемая при вызове, обнаруживает потенциально отсутствующие аннотации `Composable` там, где они были бы необходимы или ожидаемы, так что он может удобно попросить разработчика добавить их. Например: если CF вызывается внутри лямбды, компилятор дружелюбно предложит добавить `@Composable` к этой лямбде. 

Также существуют статические проверки вызовов для `@ReadOnlyComposable` CF. Они могут вызывать только другие `Composable`, предназначенные только для чтения, поскольку в противном случае мы нарушим контракт на оптимизацию, согласно которому `@ReadOnlyComposable` может только читать из композиции, но никогда не записывать в нее. Учитывая, что это должно быть выполнено на всех уровнях глубины внутри `@Composable`, нам пригодится паттерн Посетитель.

Еще одна проверка, которую мы можем найти, - это запрет на использование ссылок на функции `Composable`, поскольку на данный момент это не поддерживается JC.
#### Проверки типов
Иногда мы аннотируем типы как `Composable`, а не только функции. Для этого в компиляторе Compose есть проверка, связанная с выводом типов, чтобы он мог сообщить, когда ожидался тип, аннотированный `@Composable`, но вместо него был найден неаннотированный тип. Аналогично проверке вызовов функций, упомянутой выше. 
#### Проверки объявлений
Объявления: свойства, получатели доступа свойств, объявления функций или параметры функций, нуждаются в анализе.
Свойства, геттеры свойств и функции могут быть переопределены, даже если они аннотированы как `Composable`. В компиляторе Compose предусмотрена проверка, чтобы гарантировать, что любые переопределения любого из этих `KtElements` также аннотированы как `Composable` для сохранения согласованности.

Сюда же относится проверка, что CF не `suspend`, поскольку это не поддерживается. Такие вещи, как `main` CF или резервные поля свойств `Composable`, также запрещены.
#### Запрет проверок
Обычно это нужно, когда плагины компилятора генерируют или поддерживают код, который компилятор Kotlin обычно не принимает, чтобы обойти соответствующие проверки и заставить его работать.
Compose регистрирует `ComposeDiagnosticSuppressor` для обхода некоторых языковых ограничений, которые в противном случае привели бы к неудачной компиляции, что позволяет разблокировать некоторые специфические случаи использования.
Одно из таких ограничений касается встроенных лямбд, аннотированных "не исходными аннотациями" в местах вызова. Это аннотации с удержанием `BINARY` или `RUNTIME`. Эти аннотации сохраняются до выхода двоичных файлов, а не как аннотации `SOURCE`. Учитывая, что `inline` лямбды фактически встраиваются в вызывающие их объекты во время компиляции, они нигде не будут храниться, поэтому в этот момент аннотировать уже будет нечего. Именно поэтому Kotlin запрещает это и сообщает о следующей ошибке: "Лямбда-выражение здесь является вложенным аргументом, поэтому эта аннотация не может быть сохранена где-либо".

Это требование запрещается и для членов ожидаемых классов. Помните, что JC нацелен на мультиплатформенность, поэтому среда выполнения должна обязательно принимать `expect`-функции и свойства, помеченные как `Composable`.
#### Проверка версии среды выполнения
Первое, что происходит непосредственно перед генерацией кода, - это проверка используемой версии среды выполнения Compose. Компилятор Compose требует минимальной версии среды выполнения, поэтому в нем предусмотрена проверка, чтобы убедиться, что среда выполнения не устарела. Он способен обнаружить как отсутствие, так и устаревание среды выполнения.
Одна версия компилятора Compose может поддерживать несколько версий среды выполнения, если они выше минимально поддерживаемой. Это уже вторая проверка версий. Есть одна для версии компилятора Kotlin, а есть другая для среды выполнения JC.
### Кодогенерация
#### IR Котлин
Плагины компилятора могут не только генерировать новый код, но и модифицировать исходные тексты, поскольку они имеют доступ к IR языка до того, как из него получится конечный код для целевой платформы. Это означает, что плагин компилятора может пробраться внутрь и заменить параметры, добавить новые, изменить структуру кода перед его "фиксированием". Это происходит на бэкенд-фазах компилятора Kotlin. И, как вы, наверное, догадались, именно это делает Compose, "вставляя" неявный дополнительный параметр - `Composer` - в каждый вызов CF.

Плагины компилятора имеют возможность генерировать код в разных форматах. IR существует как представление элементов языка, которое остается независимым от целевой платформы - "промежуточное представление". Это означает, что генерация IR потенциально сделает генерируемый JC код мультиплатформенным.
Плагин компилятора Compose генерирует IR, регистрируя реализацию `IrGenerationExtension`, которая является расширением, предоставляемым общим бэкендом IR компилятора Kotlin.
#### Lowering
Термин относится к трансляции, проводимой компиляторами, с более высокого уровня или более продвинутых концепций программирования на комбинацию более атомарных концепций низкого уровня. Это довольно часто встречается в Kotlin, где у нас есть промежуточное представление (IR) языка, способное выражать довольно сложные концепции, которые затем нужно перевести в атомики более низкого уровня, прежде чем преобразовать их в байт-код JVM, Javascript, IR LLVM или любой другой платформы, на которую мы ориентируемся. Для этого в компиляторе Kotlin предусмотрена фаза lowering. Можно также понимать ее как форму нормализации.

Компилятору Compose необходимо понизить уровень некоторых концепций, поддерживаемых библиотекой, чтобы они были нормализованы до представления, понятного среде выполнения. Процесс опускания является фактической фазой генерации кода для плагина компилятора Compose. На этом этапе он обращается ко всем элементам дерева IR и произвольно изменяет IR в зависимости от потребностей среды выполнения.

Примеры того, что происходит на фазе понижения:
- Вывод стабильности класса и добавление метаданных, необходимых для его понимания во время выполнения.
- Преобразование выражений с живыми литералами таким образом, чтобы они обращались к экземплярам с изменяемым состоянием, что позволяет среде выполнения отражать изменения в исходном коде без перекомпиляции (функция живых литералов).
функция).
- Инжектирование неявного параметра Composer во все функции Composable и передача его всем вызовам CF
- Обертывание Composable тел CF для таких вещей, как:
	- Генерация различных типов групп для потока управления (заменяемые группы, перемещаемые группы...).
	- Реализация поддержки аргументов по умолчанию, чтобы они могли выполняться в пределах области видимости сгенерированной группы функции - вместо того, чтобы полагаться на стандартную поддержку параметров в Kotlin.
	- Научить функцию пропускать рекомпозиции.
	- Передача релевантной информации об изменениях состояния вниз по дереву, чтобы узлы могли автоматически пересобраться при изменении.
#### Вывод стабильности класса
Умная рекомпозиция означает пропуск рекомпозиции составных элементов, когда их входы не изменились и считаются стабильными. Стабильность - очень важная концепция в этом смысле, поскольку она означает, что CR может безопасно читать и сравнивать эти входы, чтобы пропустить рекомпозицию, когда это необходимо. Конечная цель стабильности - помочь CR.

По умолчанию стабильными считаются все примитивные типы, а также String и все типы функций. Это потому, что они неизменяемы по определению. Поскольку неизменяемые типы не изменяются, им также не нужно уведомлять композицию.

Мы также узнали, что существуют типы, которые не являются неизменяемыми, но могут считаться стабильными в Compose - для них можно использовать аннотацию `@Stable`. Одним из примеров такого типа является `MutableState`, поскольку Compose
уведомляется о каждом его изменении, поэтому можно смело полагаться на него при умной рекомпозиции.
Для пользовательских типов, которые мы создаем в нашем коде, мы можем определить, соответствуют ли они перечисленным выше свойствам, и вручную отметить их как стабильные с помощью аннотаций `@Immutable` или `@Stable`. Но полагаться на то, что разработчики будут следить за выполнением этого контракта, может быть довольно рискованно и трудно поддерживаться с течением времени. Вместо этого желательно автоматически определять стабильность класса. Compose делает это. Алгоритм вывода стабильности находится в постоянном развитии, но он выглядит следующим образом: вы посещаете каждый класс и синтезируете для него аннотацию `@StabilityInferred`. Также он
добавляет синтетическое значение `static final int $stable`, которое кодирует соответствующую информацию о стабильности класса. Это значение поможет компилятору генерировать дополнительные механизмы на последующих шагах для определения стабильности класса во время выполнения, чтобы Compose мог определить, нужно ли пересобирать наши CF, зависящие от него, или нет.

>По правде говоря, это не буквально каждый класс, а только каждый подходящий. То есть каждый публичный класс, который не является `enum`, элементом `enum`, интерфейсом, аннотацией, анонимным объектом, элементом `expect`, внутренним классом, `companion` или `inline`. Как вы уже, наверное, догадались, он также не помечен как стабильный с помощью вышеупомянутых аннотаций. Итак, в целом, это просто классы, классы данных и тому подобное, которые еще не были помечены как стабильные. Это имеет смысл, учитывая, что именно их мы будем использовать для моделирования входов наших CF

Чтобы сделать вывод о стабильности класса, Compose учитывает разные вещи. Класс будет считаться стабильным, если все параметры его конструктора также стабильны. Такие классы, как class Foo или class Foo(val value: Int), будут считаться стабильными, поскольку у них нет параметров или есть только стабильные параметры. 
Но такие вещи, как параметризованные типы в классе, также могут влиять на стабильность класса, например:
```
class Foo<T>(val value: T)
```
В этом случае `T` используется для одного из параметров класса, и поэтому стабильность Foo будет зависеть от стабильности типа, переданного для `T`. Если `T` не `reified`, он будет неизвестен до времени выполнения. Поэтому необходим механизм для определения стабильности класса во время выполнения, когда тип, переданный для `T`, известен. Чтобы решить эту проблему, компилятор Compose вычисляет и помещает в аннотацию `StabilityInferred` битовую маску, которая указывает, что вычисление стабильности данного класса во время выполнения должно зависеть от стабильности соответствующего параметра/ов типа.

Но наличие общих типов не обязательно означает нестабильность. Компилятор знает, что, например, код типа: 
```
class Foo<T>(val a: Int, b: T) { val c: Int = b.hashCode() } 
```
является стабильным, поскольку ожидается, что `hashCode` всегда будет возвращать один и тот же результат для одного и того же экземпляра. Это часть контракта.

Для классов, состоящих из других классов, стабильность определяется рекурсивно как комбинация стабильности всех аргументов. Такие вещи, как внутреннее изменяемое состояние, также делают класс нестабильным. 

В целом компилятор Compose считает тип стабильным только тогда, когда может это доказать. Например: интерфейс считается нестабильным, потому что Compose не знает, как он будет реализован.

Другой пример - типы с изменяемыми публичными свойствами, реализация которых могла бы быть неизменяемой.Такие типы также считаются нестабильными по умолчанию, поскольку компилятор не в состоянии сделать такой вывод.

Это немного не так, поскольку во многих случаях эти вещи могут быть реализованы как неизменяемые, и для CR этого должно быть достаточно. По этой причине, если модель, которую мы используем в качестве входных данных для наших CF, считается Compose нестабильной, мы все равно можем явно пометить ее как `@Stable`, если у нас есть больше информации в руках и она находится под нашим контролем
#### Включение живых литералов
Одним из флагов, которые мы можем передать компилятору, является флаг `live literals`. 
`Live literals` - это функция, при которой инструменты Compose могут отражать изменения в предварительном просмотре без необходимости перекомпиляции. Компилятор Compose заменяет эти выражения новыми версиями, которые читают свои значения из `MutableState`. Это позволяет мгновенно уведомлять среду выполнения об изменениях без необходимости перекомпиляции проекта. Как говорится в описании библиотеки `kdocs`:
*"Это преобразование предназначено для улучшения работы разработчиков и никогда не должно быть включено в релизной сборке, так как оно значительно замедлит работу кода, ориентированного на производительность"*.

Компилятор Compose генерирует уникальные идентификаторы для каждого отдельного константного выражения в нашей кодовой базе, а затем преобразует все эти константы в геттеры свойств, которые читаются из некоторого `MutableState`, хранящегося в сгенерированном классе-синглтоне для каждого файла. Во время выполнения есть API для получения значения этих констант с помощью сгенерированного ключа.
#### Мемоизация лямбд
Этот шаг генерирует удобный IR, чтобы научить среду выполнения оптимизировать выполнение лямбд, переданных CF, таких как:.
- Не-`Composable` лямбды: Компилятор генерирует IR для их мемоизации, оборачивая каждую из них в вызов `remember`. Вспомните, например, обратный вызов, который мы передаем CF.
- `Composable` лямбды: Компилятор генерирует IR для их обертывания и добавления соответствующей информации, чтобы научить среду выполнения, как хранить и читать выражение в/из Composable. Примером может служить содержимое Composable lambdas, которое мы передаем нашим узлам пользовательского интерфейса Compose при их вызове
##### Не-`Composable` лямбды
Это действие оптимизирует вызовы лямбд, передаваемые в CF, чтобы их можно было использовать повторно.
Kotlin уже оптимизирует лямбды, когда они не захватывают никаких значений, моделируя их как синглтоны, так что для всей программы существует один экземпляр, который можно использовать повторно. Такая оптимизация невозможна, когда лямбды захватывают значения, поскольку эти значения могут меняться для каждого вызова, делая его уникальным, и поэтому для каждой лямбды потребуется свой экземпляр. `Compose` более эффективен для этого конкретного случая. 

Если лямбда, которую мы передаем CF из места вызова, принимает какие-либо значения, Compose имеет возможность научить среду выполнения запоминать их. По сути, это означает обернуть его в вызов `remember`. Это делается через сгенерированный IR.Этот вызов будет запоминать лямбда-выражение на основе значений, которые он захватывает, пока эти значения стабильны. Это позволяет времени выполнения повторно использовать уже существующие лямбды вместо создания новых, пока значения, которые они захватывают, совпадают (включая входные параметры).

>Причина, по которой требуется, чтобы захваченные значения были стабильными, заключается в том, что они будут использоваться в качестве аргументов условия для вызова `remember`, поэтому они должны быть надежными для сравнения. Обратите внимание, что мемоизованные лямбды не могут быть `inline`, поскольку в противном случае было бы нечего запоминать после того, как они будут `inline` для своих вызывающих функций во время компиляции. Еще одним требованием является то, что вложенные CF не могут возвращать никаких значений, так как `remember` разрешен только в функциях, которые **эмитируют**, по соображениям дизайна.

Эта оптимизация имеет смысл только для лямбд, которые захватывают внешние значения. Если они не захватываются, то достаточно оптимизации Kotlin по умолчанию - представления их в виде синглтонов.

Как объяснялось выше, мемоизация выполняется на основе захваченных значений лямбды. При генерации IR для выражения компилятор добавит вызов `remember` с возвращаемым типом, совпадающим с типом мемоизованного выражения, затем добавит аргумент общего типа к вызову - `remember<T>...` - для соответствия типу возврата выражения. Сразу после этого он добавит все значения, захваченные лямбдой, в качестве аргументов условия - `remember<T>(arg1, arg2...)` -, чтобы их можно было использовать для сравнения, и, наконец, лямбду для выражения - `remember<T>(arg1, arg2..., expression)` - , чтобы она могла работать как концевая лямбда.

Использование захваченных значений в качестве аргументов условия гарантирует, что они будут использоваться в качестве ключей для запоминания результата выражения, поэтому оно будет считаться недействительным всякий раз, когда они изменяются.

Автоматическое запоминание лямбд, переданных CF, открывает возможности повторного использования при рекомпозиции.
##### `Composable`-лямбды
Компилятор Compose также способен обернуть все лямбды `Composable`-лямбды, чтобы научить среду выполнения, как хранить и извлекать их в/из `Composition` во время выполнения. Это еще одна форма мемоизации лямбд.

Чтобы сделать обертку, IR лямбда-выражения настраивается таким образом, что сначала вызывается фабричная функция `composable` с определенными параметрами: `composableLambda(...)`. Первым добавленным параметром будет текущий `composer`, поэтому он передается, как и ожидалось, таким образом: `composableLambda($composer, ...)`.

Затем добавляется параметр `key`, полученный из комбинации хэш-кода из полного имени `Composable`-лямбды и **начального смещения выражения**, которое, по сути, является местом его расположения в файле, чтобы убедиться, что ключ уникален - позиционная мемоизация: `composableLambda($composer, $key, ...)`. После этого добавляется булевый флаг `shouldBeTracked`. Этот параметр определяет, нужно ли отслеживать данный вызов `Composable`-лямбды  или нет. Когда у лямбд нет захватов, Kotlin превращает их в экземпляры-синглтоны, поскольку они никогда не будут меняться. Это также означает, что их не нужно отслеживать с помощью Compose.

Также может быть добавлен необязательный параметр о степени сложности выражения, необходимый только в том случае, если оно имеет более 22 параметров (магическое число): `composableLambda($composer, $key, $shouldBeTracked, $arity, ...)`

Назначение фабрики CF, вызываемой для формирования обернутого тела, простое: **добавление заменяемой группы в композицию для хранения лямбда-выражения с помощью сгенерированного ключа**. Он будет использоваться для поиска лямбды `Composable` позже. Здесь мы обучаем среду выполнения тому, как хранить и извлекать выражение `Composable` в дальнейшем. В дополнение к этой обертке `Compose` также может оптимизировать составные лямбды, которые не захватывают значения, как это делает Kotlin: Представляя их с помощью синглтонов. Для этого он генерирует синтетический `internal object` `ComposableSingletons` для каждого файла, в котором были найдены `Composable` лямбды. Этот объект будет сохранять (мемоизировать) статические ссылки на эти `Composable`-лямбды, а также включать геттеры для них, чтобы их можно было получить позже.
#### Инъектирование Composer
На этом этапе компилятор Compose заменяет все CF на их новые версии с добавлением дополнительного синтетического параметра `Composer`. Этот параметр также передается каждому вызову `Composable` в коде, чтобы он всегда был доступен в любой точке дерева. Это также включает вызовы `Composable`-лямбд.
Это также требует некоторой работы по переназначению типов, поскольку тип функции меняется, когда плагин компилятора добавляет к ней дополнительные параметры.
Это фактически делает `Composer` доступным для любого поддерева, предоставляя всю информацию, необходимую для материализации Composable-дерева и его обновления.

Встроенные лямбды, которые не являются `Composable`, намеренно не преобразуются, поскольку они исчезнут во время компиляции, когда будут встроены в вызывающие их функции. Также не преобразуются функции `expect`. Эти функции преобразуются в `actual` при разрешении типов, а значит, именно последние будут преобразованы в любом случае.
#### Распространение сравнения
Мы узнали, как компилятор вводит дополнительный параметр `$composer` и передает его во все вызовы `Composable`. Есть несколько дополнительных метаданных, которые также генерируются и добавляются в каждый `Composable`. Один из них - параметр `$changed`. Этот параметр используется для получения информации о том, изменились ли входные параметры текущего Composable с момента предыдущей композиции. Это позволяет пропустить повторные композиции.

Этот параметр синтезируется как комбинация битов, представляющих это условие **для каждого** из входных параметров функции. Для каждых n входных параметров (10 или около того) существует один параметр `$changed`, который кодирует это условие, что ограничено количеством используемых битов. Если у композита больше параметров, добавляется 2 или более флагов. Причина использования битов заключается в том, что процессоры по своей конструкции очень хорошо обрабатывают биты.

Обладание этой информацией позволяет оптимизировать время выполнения:
- Позволяет пропустить сравнения `equals`, проверяющие, изменился ли входной параметр относительно хранимого значения с предыдущих сборок. Это происходит в тех случаях, когда известно, что входной параметр статичен. Битовая маска `$changed` предоставляет эту информацию. Допустим, этот параметр является литералом `String`, как в фрагменте выше, константой или чем-то подобным. Биты этого флага сообщат CR, что значение известно во время компиляции, следовательно, оно никогда не изменится во время выполнения, и поэтому CR может не сравнивать его никогда.
- Также существует вероятность, что параметр уже известен, что он изменился с момента последней композиции. Если CR уже знает об этом заранее, то и сравнивать его нет смысла - мы и так знаем, что он изменился. В этом случае пересборка сработает наверняка, поэтому мы можем спокойно не хранить значение в таблице слотов, а просто выбросить его.
- Для всех остальных случаев состояние считается "неопределенным", поэтому время выполнения может просто продолжить, сравнить их - используя `equals` - и сохранить в таблице слотов, чтобы мы всегда могли найти последний результат позже. Значение бита для этого случая равно 0, что является случаем по умолчанию. Когда для параметра $changed передается 0, мы говорим CR, чтобы оно сделало всю работу (не использовало никаких сокращений).
Вот пример того, как выглядит тело CF после введения параметра `$changed` и необходимой логики для его обработки:
```Kotlin
@Composable  
fun Header(text: String, $composer: Composer<*>, $changed: Int) {
    var $dirty = $changed  
    if ($changed and 0b0110 === 0) {
	    $dirty = $dirty or if ($composer.changed(text)) 0b0010 else 0b0100
    if (%dirty and 0b1011 xor 0b1010 !== 0 || !$composer.skipping) { 
	    f(text) // executes body
    } else { 
	    $composer.skipToGroupEnd()
	} 
}
```
Переменная `$dirty` хранит информацию о том, изменился параметр или нет, и это определяется как битовой маской параметра `$changed`, так и, в случае необходимости, значением, ранее сохраненным в таблице слотов. Если значение считается "грязным" (изменилось), тело функции будет вызвано (пересобрано). В противном случае `Composable` пропустит рекомпозицию. Учитывая, что рекомпозиция может происходить много раз, информация о том, как изменяется входное состояние, потенциально может сэкономить времени на вычислениях.

Точно так же, как наш `Composable` получает параметр `$changed`, переданный вызывающей стороной, этот `Composable` также обязан передавать любую информацию о любых параметрах, которые у него есть, вниз по дереву. Это называется "распространение сравнения". Это информация, доступная нам в теле - во время композиции, - так что если мы уже знаем, что входной сигнал изменился, является статическим или каким-либо другим, мы можем передать эту информацию параметру `$changed` любого дочернего Composable, который повторно использует этот параметр.
#### Параметры по умолчанию
Еще одна дополнительная часть метаданных, которая добавляется к каждой CF во время компиляции, - это параметр/ы `$default`.
Поддержка аргументов по умолчанию в Kotlin не подходит для аргументов CF, поскольку CF должны выполнять выражения по умолчанию для своих аргументов внутри области видимости (порожденной группы) функции. Для этого Compose предоставляет альтернативную реализацию механизма разрешения аргументов по умолчанию.

Compose представляет аргументы по умолчанию с помощью параметра `$default`, битовой маски, сопоставляющей каждый индекс параметра с битом в маске. Это похоже на то, что сделано для параметра/ов `$changed`. Также существует один параметр `$default` на каждые n входных параметров со значениями по умолчанию. Эта битовая маска предоставляет информацию о том, есть ли у параметров значение, указанное в месте вызова, или нет, чтобы определить, нужно ли использовать выражение по умолчанию.
#### Поток управления генерацией групп
Компилятор Compose также вставляет группу в тело каждой CF. 

В конечном итоге CF эмитируют группы во время выполнения, и эти группы содержат всю необходимую информацию о текущем состоянии Composable-вызова. Это позволяет Composition знать, как очистить все записанные данные, когда группу нужно заместить (замещаемые группы), переместить данные, сохраняя идентичность `Composable` все время (перемещаемые группы), или перезапустить функцию во время рекомпозиции (перезапускаемые группы).

В конце концов, время выполнения должно знать, как работать с потоком управления, основываясь на информации, которую Композиция хранит в памяти. 

Группы также несут информацию о позиции вызова в источниках. Они оборачивают участок текста в источниках и имеют ключ, сгенерированный с использованием позиции вызова в качестве одного из факторов. Это позволяет хранить группу и дает возможность позиционной мемоизации.
##### Замещаемые группы
Несколько разделов назад мы объяснили, что тело лямбды `Composable` автоматически оборачивается путем вставки вызова фабрики функций Composable, которой передается такая информация, как `$composer`, сгенерированный `$key` и собственно выражение лямбды `Composable`, а также многое другое.
Вот как эта фабричная функция выглядит в коде:
```Kotlin
fun composableLambda(
composer: Composer,
key: Int,
tracked: Boolean,
block: Any
): ComposableLambda {
	composer.startReplaceableGroup(key)
	val slot = composer.rememberedValue()
	val result = if (slot === Composer.Empty) {
		val value = ComposableLambdaImpl(key, tracked)
		composer.updateRememberedValue(value)
		value
	} else {
		slot as ComposableLambdaImpl
	}
	result.update(block)
	composer.endReplaceableGroup()
	return result
}
```
Эта фабричная функция вызывается для Composable-лямбд, подобных тем, что используются для содержимого наших CF. Если мы внимательно посмотрим на нее, то заметим, что она начинает замещаемую группу с ключа первой, а закрывает группу в конце, заворачивая весь текстовый диапазон в середину. Между вызовами начала и конца он обновляет композицию соответствующей информацией. Для данного конкретного случая это и есть лямбда-выражение, которое мы оборачиваем. Это касается лямбд Composable, но для других вызовов Composable все происходит точно так же. 

Группы похожи на дерево. Каждая может содержать любое количество дочерних. Если вызов CF также является `Composable`, компилятор также вставит для него группу.

Функция `Composable`, выполняющая некоторую условную логику, также будет испускать замещаемую группу, так что она хранит группу, которую можно заместить позже, когда условие изменится.
##### Перемещаемые группы
Могут быть переупорядочены без потери идентичности. Например, требуются для вызовов `key`:
```Kotlin
@Composable
fun TalksScreen(talks: List<Talk>)
	Column {  
		for (talk in talks) {
			key(talk.id) { // Unique key Talk(talk)
				Talk(talk)
			}
		}
	}
}
```
Обертывание нашего `Talk` в `key`гарантирует, что ему будет присвоена уникальная идентичность без исключений. Когда мы оборачиваем CF ключом, создается перемещаемая группа. Это поможет изменить порядок вызовов без риска потерять идентичность элементов.
Вот пример того, как преобразуется Composable с использованием `key`:
```Kotlin
// Before compiler (sources)
@Composable  
fun Test(value: Int) {
	key(value) {
	    Wrapper {
	      Leaf("Value ${'$'}value")
	    }
	}
}

// After
@Composable  
fun Test(value: Int, %composer: Composer?, %changed: Int) {
	// ...
	%composer.startMovableGroup(<>, value)
	Wrapper(
		composableLambda(%composer, <>, true) {%composer: Composer?,
	%changed: Int -> Leaf("Value %value", %composer, 0) },
		%composer, 
		0b0110) 
	%composer.endMovableGroup()  
// ...
}
```
##### Перезапускаемые группы
Наиболее интересными, вероятно, являются перезапускаемые группы. Они вставляются только для перезапускаемых CF. Они также оборачивают соответствующий вызов `Composable`, но поверх этого немного расширяют конечный вызов, чтобы он возвращал `null`. Это значение будет `null`, только когда тело вызова CF не считывает никаких состояний, которые могут измениться, поэтому рекомпозиция никогда не потребуется. В этом смысле нет необходимости обучать среду выполнения тому, как рекомпозировать этот `Composable`. В противном случае, если он возвращает ненулевое значение, компилятор генерирует лямбду, которая учит среду выполнения, как "перезапустить" (повторно выполнить) `Composable` и, следовательно, обновить Composition.
Вот как это выглядит в коде:
```Kotlin
// Before compiler (sources)

@Composable fun A(x: Int) { 
	f(x)
} 
// After compiler
@Composable  
fun A(x: Int, $composer: Composer<*>, $changed: Int) {
	$composer.startRestartGroup()  
	// ...  
	f(x)  
	$composer.endRestartGroup()?.updateScope { next ->
    A(x, next, $changed or 0b1)
  }
}
```
Именно такой тип группы генерируется для всех CF, которые читают из состояния.
Из официальной документации:
- Если блок выполняется ровно 1 раз всегда, то группы не нужны.
- Если набор блоков таков, что ровно один из них выполняется одновременно (например, результат блока оператора if или условия when), то мы вставляем заменяемую группу вокруг каждого блока. У нас есть условная логика.
