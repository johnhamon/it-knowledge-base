### Структура Composer
Итак, что же это за контекст вызова, который мы передаем друг другу, и зачем нам это нужно?

Мы называем этот объект `Composer`. Реализация `Composer` содержит структуру данных, которая тесно связана со структурой данных [[Буферное окно]]. Она широко используется в текстовых редакторах.
![[slot table 1.jpg]]
Теперь исполняемая иерархия `Composable` может обращаться к этой структуре данных и вставлять в нее данные.
![[slot table 2.jpg]]
Представим, что мы закончили выполнение иерархии. В какой-то момент нам нужно снова что-то рекомпозировать. Поэтому мы сбрасываем курсор на вершину массива и снова выполняем процедуру. В процессе выполнения мы ничего не делать или обновить значение.
![[slot table 3.jpg]]
Мы можем решить, что структура пользовательского интерфейса изменилась, и захотим сделать вставку. В этот момент мы перемещаем пробел в текущую позицию.
![[slot table 4.jpg]]
Теперь можем производить вставку:
![[slot table 5.jpg]]
Причина, по которой мы выбрали эту структуру данных, в том, что мы делаем ставку на то, что в среднем UI не очень часто меняет структуру. Когда у нас есть динамический UI, он меняет свои значения, но структуру - значительно реже. Когда структура меняется, она, как правило, меняется большими кусками, так что выполнение этого O(n)-промежуточного перемещения является разумным компромиссом.

Рассмотрим счетчик
```Kotlin
@Composable
fun Counter() {
	var count by remember { mutableStateOf(0) }
	Button(
		text = "Count: $count",
		onPress = { count += 1 }
	)
}
```
Как выглядит добавление компилятором Compose `Composer`(компилятор также передает объект `$composer` во все вызовы `Composable` в теле функции.):
```Kotlin
fun Counter($composer: Composer) {
	$composer.start(123)
	var count by remember($composer) { mutableStateOf(0) }
	Button(
		$composer,
		text = "Count: $count",
		onPress = { count += 1 }
	)
	$composer.end()
}
```
Когда этот `Composer` выполняется, он делает следующее:

- вызывается `Composer.start `и сохраняет объект[[Компилятор Compose#Поток управления генерацией групп | группы]] 
- `remember` вставляет объект группы
- сохраняется значение, которое возвращает `mutableStateOf` - экземпляр состояния.
- `Button` сохраняет группу, а затем каждый из ее параметров. 

И наконец, мы приходим к `composer.end`.
![[slot table 6.jpg]]
Теперь в структуре данных хранятся все объекты из композиции, все дерево в порядке выполнения, то есть фактически обход дерева в глубину.

Теперь все эти групповые объекты занимают много места, так для чего же они нужны? Эти групповые объекты предназначены для управления перемещениями и вставками, которые могут происходить в динамическом UI. Компилятор знает, как выглядит код, изменяющий структуру UI, поэтому он может условно вставить эти группы. Чаще всего они не нужны компилятору, поэтому он не вставляет столько групп в таблицу слотов. Чтобы проиллюстрировать это, посмотрите на следующую условную логику.
```Kotlin
@Composable 
fun App() {
	val result = getData()
	if (result == null) {
		Loading(...)
	} else {
		Header(result)
		Body(result)
	}
}
```
В этом `Composable` функция `getData` возвращает некоторый результат и выдает `Loading` в одном случае, а в другом - `Header` и `Body`. Компилятор вставляет отдельные ключи для каждой ветви оператора `if`.
```Kotlin
fun App($composer: Composer) {
	val result = getData()
	if (result == null) {
	   $composer.start(123)
	   Loading(...)
	   $composer.end()
	} else {
	   $composer.start(456)
	   Header(result)
	   Body(result)
	   $composer.end()
 }
}
```
Предположим, что при первом выполнении этого кода результат будет равен `null`. Это вставляет группу в массив пробелов и запускает экран загрузки.
![[slot table 7.jpg]]
При втором запуске функции предположим, что результат уже не равен нулю, поэтому выполняется вторая ветвь `if`. Вот тут-то все и становится интересным.

В вызове `composer.start` есть группа с ключом 456. Компилятор видит, что группа в таблице слотов 123 не совпадает, поэтому теперь он знает, что структура UI изменилась.

Затем компилятор перемещает пробел в текущую позицию курсора и расширяет пробел на весь UI, который там был, фактически избавляясь от него.

В этот момент код выполняется как обычно, и вставляется новый UI - заголовок и тело.
![[slot table 8.jpg]]
В данном случае накладными расходами оператора `if` была запись одного слота в таблице слотов. Вставляя эту единственную группу, мы получаем произвольный поток управления в нашем UI, позволяя компилятору управлять им и обращаться к этой кэш-подобной структуре данных во время выполнения UI.

Эту концепцию мы называем позиционной мемоизацией, и именно на ней с нуля построен JC.
### Позиционная мемоизация
Обычно мы используем глобальную мемоизацию, означающую, что компилятор кэширует результат функции на основе входов этой функции. Чтобы проиллюстрировать пример позиционной мемоизации, возьмем эту CF, которая выполняет вычисления.
```Kotlin
@Composable
fun App(items: List<String>, query: String) {
	 val results = items.filter { it.matches(query) }
	 // ...
}
```
Функция принимает список строковых элементов и запрос, а затем выполняет вычисление фильтра над списком. Мы можем обернуть это вычисление в вызов `remember`: `remember` - это нечто, что знает, как обратиться к таблице слотов. `remember` просматривает элементы и сохраняет список и запрос в таблице слотов. Затем выполняется вычисление фильтра, и `remember` сохраняет результат, а затем передает его обратно.

При втором выполнении функции `remember` смотрит на новые передаваемые значения и сравнивает их со старыми. Если ни одно из них не изменилось, то операция фильтрации пропускается и возвращается предыдущий результат. Это и есть позиционная мемоизация.

Интересно, что эта операция очень дешевая: компилятору нужно было сохранить одно предыдущее обращение. Это вычисление может происходить по всему UI, и, поскольку вы храните его позиционно, оно сохраняется только для этого места.

Но есть интересный вырожденный случай, когда входные данные равны нулю. Одна из вещей, которую мы можем сделать, это намеренно злоупотребить этим API. Мы можем мемоизировать заведомо грязное вычисление, например Math.random.
```Kotlin
@Composable 
fun App() {
	val x = remember { Math.random() }
	// ...
}```
Если бы вы делали это с помощью глобальной мемоизации, это не имело бы смысла. Но при позиционной мемоизации это приобретает новую семантику. Каждый раз, когда мы используем App в иерархии CF, возвращается новое значение Math.random. Однако каждый раз, когда композит перекомпонуется, будет возвращаться одно и то же значение Math.random. Это порождает персистентность, а персистентность порождает состояние.
### Хранение параметров
Чтобы проиллюстрировать, как хранятся параметры CF, давайте рассмотрим CF, которая принимает число, вызывает другую CF и рендерит адрес.
```Kotlin
@Composable 
fun Google(number: Int) {
	Address(
		number=number,
		street="Amphitheatre Pkwy",
		city="Mountain View",
		state="CA"
		zip="94043"
	)
}
 
@Composable 
fun Address(
	number: Int,
	street: String,
	city: String,
	state: String,
	zip: String
) {
	Text("$number $street")
	Text(city)
	Text(", ")
	Text(state)
	Text(" ")
	Text(zip)
}
```
Compose хранит параметры композитной функции в таблице слотов. Если это так, то, глядя на пример выше, мы видим некоторые излишества: "Mountain View" и "CA", которые добавляются в вызове адреса, снова сохраняются в вызовах текста, поэтому эти строки будут сохранены дважды.

Мы можем избавиться от этой избыточности, добавив параметр `$static` к CF на уровне компилятора. В данном случае - это битовое поле, указывающее, знает ли среда выполнения, что параметр не изменяется. Если известно, что параметр не меняется, то хранить его не нужно. Таким образом, в этом примере Google компилятор передает битовое поле, которое говорит, что ни один из параметров не изменится.

Затем, в `Address`, компилятор может сделать то же самое и передать это в текст.
```Kotlin
fun Address(
  $composer: Composer,
  $static: Int,
  number: Int, street: String, 
  city: String, state: String, zip: String
) {
  Text($composer, ($static and 0b11) and (($static and 0b10) shr 1), "$number $street")
  Text($composer, ($static and 0b100) shr 2, city)
  Text($composer, 0b1, ", ")
  Text($composer, ($static and 0b1000) shr 3, state)
  Text($composer, 0b1, " ")
  Text($composer, ($static and 0b10000) shr 4, zip)
}```
Эта побитовая логика трудночитаема и запутанна, но нам и не нужно ее понимать: компиляторы в этом разбираются, а люди - нет.

В примере с Google мы видим, что информация избыточна, но здесь есть и константы. Оказывается, хранить их тоже не нужно. Таким образом, вся иерархия определяется параметром `number`, и это единственное значение, которое компилятору нужно хранить.

Благодаря этому мы можем пойти дальше и сгенерировать код, который понимает, что число - это единственное, что будет меняться. Этот код может работать так, что если число не изменилось, то тело функции будет пропущено, и мы можем просто указать компилятору переместить текущий индекс туда, где он будет находиться, как если бы функция выполнилась.
```Kotlin
fun Google(
 $composer: Composer,
 number: Int
) {
 if (number == $composer.next()) {
   Address(
     $composer,
     number=number,
     street="Amphitheatre Pkwy",
     city="Mountain View",
     state="CA"
     zip="94043"
   )
 } else {
   $composer.skip()
 }
}```
`Composer` знает, насколько нужно перемотать вперед выполнение, чтобы возобновить его в нужный момент.
### Рекомозиция
Вернемся к примеру со счетчиком. Сгенерированный код, который компилятор создает для этого счетчика, имеет `composer.start` и `compose.end`. При выполнении `Counter` среда выполнения понимает, что когда она вызывает `count.value`, то читает свойство экземпляра appmodel. Во время выполнения, когда мы вызываем `compose.end`, мы по желанию возвращаем значение.
```Kotlin
fun Counter($composer: Composer) {
	$composer.start(123)
	var count by remember($composer) { mutableStateOf(0) }
	Button(
		$composer,
		text = "Count: $count",
		onPress = { count.value += 1 }
	)
	$composer.end()
}
```
```Kotlin
$composer.end()?.updateScope { nextComposer ->
 Counter(nextComposer)
}
```
Затем мы можем вызвать метод `updateScope` для этого значения с лямбдой, которая сообщает среде выполнения, как перезапустить этот `Composable`, если это необходимо. Это эквивалентно лямбде, которую в противном случае получил бы `LiveData`. Причина, по которой здесь стоит знак вопроса - причина, по которой это значение является нулевым, - заключается в том, что если мы не читаем никаких объектов модели во время выполнения `Counter`, то нет причин учить среду выполнения, как обновить это значение, потому что мы знаем, что оно никогда не будет обновляться.
