### Таблица слотов и список изменений
#### Таблица слотов
Таблица слотов - это оптимизированная структура в памяти, которую среда выполнения использует для хранения текущего состояния композиции. Она заполняется данными при начальной композиции и обновляется при каждой рекомпозиции. Мы можем думать о ней как о трассировке всех вызовов функций `Composable`, включая их расположение в источниках, параметры, запомненные значения, `CompositionLocals` и многое другое. Все, что происходило во время композиции, находится там. Вся эта информация используется `Composer` для создания следующего списка изменений, поскольку любые изменения в дереве всегда будут зависеть от текущего состояния Композиции.

В то время как таблица слотов записывает состояние, список изменений - это то, что вносит фактические изменения в дерево узлов. Его можно понимать как файл патча, который после применения обновляет дерево. Все изменения необходимо записать, а затем применить. За применение изменений в списке отвечает `Applier`, который является абстракцией, на которую опирается среда выполнения, чтобы в конечном итоге материализовать дерево. 

Наконец, `Recomposer` координирует все это, решая, когда и в каком потоке рекомпозировать, и когда и в каком потоке применить изменения.
#### Глубокое рассмотрение таблицы слотов
Таблица слотов - это структура данных, оптимизированная для быстрого линейного доступа. Она основана на идее "буферного окна", очень распространенной в текстовых редакторах. По этой причине данные в ней хранятся в двух линейных массивах. В одном из этих массивов хранится информация о группах, доступных в Композиции, в другом - слоты, принадлежащие каждой группе
```Kotlin
var groups = IntArray(0) 
	private set
var slots = Array<Any?>(0) { null } 
	private set
```
>CF эмитируют группы, которые задают идентификацию `Composable` после его сохранения в памяти (уникальный ключ), чтобы впоследствии его можно было идентифицировать. Группы содержат всю необходимую информацию о вызове `Composable` и его дочерних элементах, а также информацию о том, как обращаться с `Composable` (как с группой). Группы могут иметь различный тип в зависимости от шаблонов потока управления, найденных в теле CF: перезапускаемые, перемещаемые, замещаемые, переиспользуемые и т.д.

Массив `groups` использует `Int`, поскольку в нем будут храниться только "поля групп", которые представляют собой метаданные для них. Родительские и дочерние группы хранятся в виде групповых полей. Поскольку это линейная структура данных, групповые поля родительской группы всегда будут на первом месте, а за ними последуют групповые поля всех ее дочерних групп. Это линейный способ моделирования дерева групп, и он благоприятствует линейному сканированию дочерних групп. Случайный доступ является дорогостоящим, если только он не осуществляется через якорь группы. `Anchors` похожи на указатели, которые существуют по этой причине.

С другой стороны, массив слотов хранит соответствующие данные для каждой из этих групп. В нем хранятся значения любого типа (`Any?`), поскольку он предназначен для хранения информации любого типа. Именно здесь хранятся фактические данные о составе. Каждая группа, хранящаяся в массиве `groups`, описывает, как найти и интерпретировать ее слоты в массиве `slots`, поскольку группа всегда связана с диапазоном слотов.

Для чтения и записи таблица слотов опирается на зазор (gap). Думайте о нем как о диапазоне позиций из таблицы. Этот промежуток перемещается и определяет, откуда считываются и куда записываются данные в массивах, когда приходит время. Зазор имеет указатель, указывающий, с чего начать запись, и может сдвигать свои начальную и конечную позиции, поэтому данные в таблице также могут быть перезаписаны
![[slot table image.jpg]]

```Kotlin
@Composable 
@NonRestartableComposable 
fun ConditionalText() {
	if (a) { 
		Text(a)
	} else { 
		Text(b) 
	}
}
```
Если этот `Composable` помечен как `@NonRestartabale`, будет вставлена замещаемая группа (вместо перезапускаемой). Группа будет хранить данные в таблице для текущего "активного" ребенка. Это будет `Text(a)`, в случае если `a` истинно. Когда условие переключится, пробел переместится обратно в начальную позицию группы, и она начнет запись оттуда, перекрывая все слоты данными для `Text(b)`.

Для чтения из таблицы и записи в нее у нас есть `SlotReader` и `SlotWriter`. В таблице слотов может быть несколько активных читателей, но один активный писатель. После каждой операции чтения или записи соответствующий читатель или писатель закрывается. Может быть открыто любое количество читателей, но для безопасности таблицу можно читать только тогда, **когда в нее не ведется запись**. Таблица `SlotTable` остается недействительной до тех пор, пока активный писатель не будет закрыт, поскольку он будет напрямую изменять группы и слоты, что может привести к условиям гонки, если мы попытаемся читать из нее в то же самое время.

`SlotReader` работает как Посетитель. Он отслеживает текущую группу, которая читается из массива `groups`, ее начальную и конечную позиции, ее родителя (хранится непосредственно перед ней), текущий слот из читаемой группы, количество слотов в группе... и т. д. `SlotReader` может переставлять, пропускать группы, считывать значение из текущего слота, считывать значения из определенных индексов и т.п. Другими словами, он используется для чтения информации о группах и их слотах из массивов.

`SlotWriter`, с другой стороны, используется для записи групп и слотов в массивы. Как объяснялось выше, он может записывать в таблицу данные любого типа (`Any?`). `SlotWriter` полагается на упомянутые выше пробелы для групп и слотов, поэтому он использует их для определения места записи (позиции) в массивах.

Думайте о зазоре как о **сдвигаемом и изменяемом по размеру промежутке позиций** для линейного массива. Писатель отслеживает начальную и конечную позиции, а также длину каждого зазора. Он может перемещать пробел, обновляя его начальную и конечную позиции.

Читатель может добавлять, заменять, перемещать и удалять группы и слоты. Например, можно добавить новый узел `Composable` в дерево или `Composables` в условной логике, которые нужно заменить при переключении условий. Писатель может пропускать группы и слоты, продвигаться на заданное количество позиций, стремиться к позиции, определенной `Anchor`, и выполнять множество других подобных операций.

Для быстрого доступа к таблице он хранит список якорей, указывающих на определенные индексы. Позиция каждой группы - также называемая индексом группы - в таблице также отслеживается через якорь. Якорь обновляется, когда группы перемещаются, заменяются, вставляются или удаляются до позиции, на которую указывает якорь.

Таблица слотов также работает как итератор групп композиций, поэтому она может предоставлять информацию о них инструментам, чтобы те могли проверять и представлять детали композиции.
#### Список изменений 
Мы узнали о таблице слотов, о том, как она позволяет среде  выполнения отслеживать текущее состояние `Composition`. Верно, но какова же тогда роль списка изменений? Когда он создается? Что он моделирует? Когда эти изменения применяются и по какой причине? 

Каждый раз, когда происходит композиция (или рекомпозиция), CF из наших источников выполняются и эмитируют. Эмиссия означает создание отложенных изменений для обновления таблицы слотов, а в конечном итоге и материализованного дерева. Эти изменения хранятся в виде списка. Формирование этого свежего списка изменений основано на том, что уже хранится в таблице слотов. Помните: любые изменения в дереве должны зависеть от текущего состояния Композиции.

Примером может служить перемещение узла. Представьте, что вы изменили порядок компонент списка. Нам нужно проверить, где этот узел был расположен в таблице раньше, удалить все эти слоты и записать их снова, но уже с новой позиции.

Другими словами, каждый раз, когда Composable эмитирует, он просматривает таблицу слотов, создает отложенное изменение в соответствии с потребностями и текущей доступной информацией и добавляет его в список со всеми изменениями. Позже, когда композиция будет завершена, наступит время материализации, и эти записанные изменения будут действительно выполнены. Именно тогда они действительно обновят таблицу слотов самой свежей доступной информацией о композации. Именно этот процесс делает процесс эмиссии очень быстрым: Он просто создает отложенное действие, которое будет ожидать выполнения.

Далее мы видим, что список изменений - это то, что в конечном итоге вносит изменения в таблицу. Сразу после этого он уведомит `Applier` об обновлении материализованного дерева узлов. Как мы уже говорили, `Recomposer` организует этот процесс и решает, в каком потоке собирать или пересобирать, а в каком - применять изменения из списка. Последний также будет контекстом по умолчанию, используемым `LaunchedEffect` для запуска эффектов.

Таким образом, мы получили более четкое представление о том, как изменения записываются, откладываются и в конечном итоге выполняются, а также как все состояние хранится в таблице слотов. Теперь самое время познакомиться с `Composer`.
### The Composer
Давайте рассмотрим, как добавляются узлы в представление дерева в памяти. Для примера мы можем использовать компоновку Layout Composable. Layout - это совокупность всех компонентов пользовательского интерфейса, предоставляемых Compose UI. Вот как это выглядит в коде:
```Kotlin
@Suppress("ComposableLambdaParameterPosition") 
@Composable inline fun Layout(
  content: @Composable () -> Unit,
  modifier: Modifier = Modifier,
  measurePolicy: MeasurePolicy
  ) {
	val density = LocalDensity.current  
	val layoutDirection = LocalLayoutDirection.current
	ReusableComposeNode<ComposeUiNode, Applier<Any>>(
		factory = ComposeUiNode.Constructor,
		update = {
			set(measurePolicy, ComposeUiNode.SetMeasurePolicy) 
			set(density, ComposeUiNode.SetDensity) 
			set(layoutDirection, ComposeUiNode.SetLayoutDirection)
		},
	    skippableUpdate = materializerOf(modifier),
	    content = content
	) 
}
```
`Layout` использует `ReusableComposeNode` для эмиссии `LayoutNode` в композицию. Но даже если это звучит как создание и добавление узла сразу, на самом деле это обучение среды выполнения тому, как создать, инициализировать и вставить узел в текущее место в композиции, когда придет время. 
```Kotlin
@Composable  
inline fun <T, reified E : Applier<*>> ReusableComposeNode(
	noinline factory: () -> T,  
	update: @DisallowComposableCalls Updater<T>.() -> Unit,  
	noinline skippableUpdate: @Composable SkippableUpdater<T>.() -> Unit, 
	content: @Composable () -> Unit
){
	// ...
	currentComposer.startReusableNode()
	// ...
	 currentComposer.createNode(factory)
	// ...
	Updater<T>(currentComposer).update() // initialization
	// ...
	currentComposer.startReplaceableGroup(0x7ab4aae9)  
	content() 
	currentComposer.endReplaceableGroup() 
	currentComposer.endNode()
}
```
Все делегируется экземпляру `сurrentComposer`. Мы также видим, как он использует возможность создать заменяемую группу, чтобы обернуть содержимое этого `Composable` при его сохранении. Любые дочерние элементы, испускаемые в лямбде `content`, фактически будут сохранены как дочерние элементы этой группы (а значит, и `Composable`) в `Composition`. Та же операция эмиссии выполняется для любых других функций `Composable`.
### Моделирование изменений
Все операции по выдаче, делегированные `currentComposer`, внутренне моделируются как `Changes`, которые добавляются в список. `Change` - это отложенная функция с доступом к текущему `Applier` и `SlotWriter` (помните, что одновременно активен только один пишущий механизм). Посмотрим на это в коде:
```Kotlin
internal typealias Change = ( 
	applier: Applier<*>,  
	slots: SlotWriter, 
	rememberManager: RememberManager
) -> Unit
```
Эти изменения добавляются в список (записываются). Действие "эмиссии" по сути означает создание этих изменений, которые представляют собой отложенные лямбды для потенциального добавления, удаления, замены или перемещения узлов из таблицы слотов и, соответственно, уведомления `Applier` (чтобы эти изменения могли быть материализованы).

По этой причине всякий раз, когда мы говорим об "эмиссии изменений", мы также можем использовать слова "запись изменений" или "планирование изменений". Все это означает одно и то же.

После композиции, когда все вызовы функций Composable завершены и все изменения записаны, все они будут **пакетно применены Applier-ом**
#### Оптимизация записи
Как мы узнали выше, вставка новых узлов делегируется `Composer`. Это означает, что он всегда знает, когда уже погружен в процесс вставки новых узлов в композицию. В этом случае `Composer` может сократить процесс и начать запись в таблицу слотов сразу же после появления изменений, вместо того чтобы записывать их (добавлять в список для последующей интерпретации). В другом случае эти изменения записываются и откладываются, поскольку время для их внесения еще не пришло.
#### Чтение и запись групп
Когда композиция завершена, вызывается `composition.applyChanges()`, чтобы материализовать дерево, и изменения записываются в таблицу слотов. `Composer` может записывать различные типы информации: данные, узлы или группы. Однако для простоты все они в конечном итоге хранятся в виде групп. Они просто имеют разные групповые поля для различия.

Композитор может "стартовать" и "закончить" любую группу. Это имеет разные значения в зависимости от выполняемых действий. Если это запись, то это будет означать "группа создана" и "группа удалена" из таблицы слотов. Если это чтение, то `SlotReader` будет попрошено переместить указатели чтения в группу или из группы, чтобы начать или закончить чтение из нее.

Узлы в дереве `Composable` (в конечном счете группы в таблице) не только вставляются, но и могут быть удалены или перемещены. Удаление группы означает удаление ее и всех соответствующих слотов из таблицы. Для этого `Composer` просит соответствующим образом переместить `SlotReader` и заставить его пропустить группу (поскольку ее там больше нет), а также записать операции по удалению всех ее узлов из `Applier`. Любые действия по модификации должны быть запланированы (записаны) и применены позже в виде пакета, как объяснялось выше, в основном для того, чтобы убедиться, что все они имеют смысл вместе. `Composer` также отбросит все ожидающие аннулирования для удаленной группы, поскольку они никогда не произойдут.

>Не все группы могут быть перезапускаемыми, замещаемыми, перемещаемыми или многократно используемыми. Среди прочих вещей, которые также хранятся в виде групп, мы можем найти блок-обертку для значений по умолчанию. Этот блок окружает запоминаемые значения для вызовов `Composable`, необходимые для создания параметров по умолчанию: `model: Model = remember { DefaultModel() }.` Это также хранится как очень специфическая группа.

Когда `Composer` хочет создать группу, происходят следующие вещи:
- Если `Composer` находится в процессе вставки значений, он продолжит запись в таблицу слотов, поскольку нет причин ждать.
- В другом случае, если есть незавершенные операции, он запишет эти изменения в применяемые при применении изменений. Здесь `Composer` попытается повторно использовать группу, если она уже существует (в таблице).
- Если группа уже хранится, но находится в другом месте (была перемещена), будет записана операция перемещения всех слотов для группы.
- Если группа новая (не найдена в таблице), она перейдет в режим вставки, который запишет группу и все ее дочерние элементы в промежуточную таблицу `insertTable` (еще одну `SlotTable`), пока группа не будет завершена. Это позволит запланировать вставку групп в конечную таблицу.
- Наконец, если `Composer` не занимается вставкой и нет незавершенных операций записи, он попытается начать чтение группы.
Повторное использование групп - обычное дело. Иногда нет необходимости создавать новый узел, но мы можем повторно использовать его, если он уже существует. (См. `ReusableComposeNode` выше). При этом будет выполнена (записана) операция перехода к узлу со стороны `Applier`, но будут пропущены операции по его созданию и инициализации. Когда свойство узла нуждается в обновлении, это действие также записывается как `Change`.
#### Запоминание значений
Мы узнали, что `Composer` имеет возможность запоминать значения в композиции (записывать их в таблицу слотов), а также обновлять их впоследствии. Сравнение, чтобы проверить, изменилось ли оно с прошлого состава, выполняется прямо при вызове `remember`, но действие обновления записывается как `Change`, если только Composer уже не вставляет значение.
Если значение для обновления является `RememberObserver`, то `Composer` также запишет неявное `Change`, чтобы отследить действие запоминания в `Composition`. Это понадобится позже, когда все эти запомненные значения нужно будет забыть.
#### Рекомпозиция зон видимости
Еще одним элементом, который также происходит через `Composer`, являются зоны видимости `recompose`, которые позволяют выполнять интеллектуальную рекомпозицию. Они напрямую связаны с группами перезапуска. Каждый раз, когда создается группа перезапуска, `Composer` создает для нее `RecomposeScope` и устанавливает его в качестве текущего `RecomposeScope` для композиции. 
`RecomposeScope` моделирует область Композиции, которая может быть перекомпонована независимо от остальной части Композиции. Он может использоваться для ручного аннулирования и запуска рекомпозиции `Composition`. Запрос на аннулирование осуществляется через композитор, например: `composer.currentRecomposeScope().invalidate()`. Для перекомпоновки композитор позиционирует таблицу слотов в начальное местоположение этой группы, а затем вызывает блок `recompose`, переданный в команду лямбда. Это приведет к повторному вызову функции `Composable`, которая испустит еще один сигнал и, следовательно, попросит `Composer` переопределить существующие данные в таблице.

`Composer` ведет стек всех областей `recompose`, которые были признаны недействительными. Это означает, что они ожидают перекомпоновки, или, другими словами, должны быть вызваны при следующей перекомпоновке. Текущий `RecomposeScope` можно получить, заглянув в этот стек.

При этом `RecomposeScopes` не всегда включены. Это происходит только тогда, когда `Compose` обнаруживает операции чтения из снимков `State` внутри `Composable`. В этом случае `Composer` помечает `RecomposeScope` как использованный, что заставляет вставленный вызов "end" в конце Composable `больше не возвращать null`, а значит, активировать следующую за ним лямбду рекомпозиции (см. ниже, после символа ?).
```Kotlin
// After compiler inserts boilerplate
@Composable  
fun A(x: Int, $composer: Composer<*>, $changed: Int) {
	$composer.startRestartGroup()  
	// ...  
	f(x)  
	$composer.endRestartGroup()?.updateScope { next ->
		A(x, next, $changed or 0b1)
	}
}
```
`Composer` может перекомпоновать все недействительные дочерние группы текущей родительской группы, если перекомпоновка необходима, или просто заставить читателя пропустить группу до конца, если это не так. 
#### Побочные эффекты в Composer
`Composer` также может записывать `SideEffects`. Побочный эффект всегда запускается **после композиции**. Они записываются в виде функции, которую нужно вызвать, когда изменения в соответствующем дереве уже применены. Этот тип эффектов совершенно не зависит от жизненного цикла `Composable`. Мы не получим таких вещей, как автоматическая отмена при выходе из `Composition` или перезапуск эффектов при рекомпозиции. Это связано с тем, что данный тип эффектов не хранится в таблице слотов, а значит, просто отбрасывается при сбое композиции.
#### Хранение CompositionLocals
`Composer` также предоставляет средства для регистрации `CompositionLocals` и получения их значений по ключу. Вызовы `CompositionLocal.current` полагаются на это. Провайдер и его значения также хранятся как группа в таблице слотов, все вместе.
#### Хранение исходной информации
`Composer` также хранит исходную информацию в виде `CompositionData`, собранных во время создания композиции, для использования инструментами Compose.
#### Связывание композиций через CompositionContext
Существует не одна композиция, а дерево композиций и подкомпозиций. Подкомпозиции - это композиции, созданные  встроенными только для того, чтобы построить отдельную композицию в контексте текущей для поддержки независимой инвалидации.

Подкомпозиция связана со своей родительской композицией с помощью ссылки на родительский `CompositionContext`. Этот контекст существует для того, чтобы связывать композицию и подкомпозиции вместе в виде дерева. Он гарантирует, что `CompositionLocals` и инвалидации прозрачно разрешаются/распространяются вниз по дереву, как если бы они принадлежали одной композиции. Сам `CompositionContext` также записывается в таблицу слотов как группа.

Создание субкомпозиций обычно выполняется через 
```Kotlin
@Composable fun rememberCompositionContext(): CompositionContext { return currentComposer.buildContext()
```
Эта функция запоминает новую композицию на текущей позиции в таблице слотов или возвращает ее, если она уже запомнена. Она используется для создания субкомпозиции из мест, где требуется отдельная композиция, таких как `VectorPainter`, `Dialog`, `SubcomposeLayout`, `Popup` или собственно `AndroidView`, который является оберткой для интеграции Android Views в Composable trees.
#### Доступ к снимку текущего состояния
В `Composer` есть ссылка на текущий снимок, то есть снимок значений, возвращаемых изменяемыми состояниями и другими объектами состояния для текущего потока. Все объекты состояния будут иметь в снимке то же значение, что и при создании снимка, если только они не были явно изменены в нем.
### Навигация по дереву узлов 
Осуществляется `applier`-ом, но не напрямую. Она осуществляется путем записи всех местоположений узлов по мере их обхода читателем и записи их в массив `downNodes`. Когда навигация по узлам реализована, все `downs` в узлах `downNodes` воспроизводятся аппликанту. Если узел `up` записывается до того, как будет реализован соответствующий узел `down`, то он просто удаляется из стека downNodes в качестве сокращения.
#### Обеспечение синхронизации читателя и писателя
Это немного низкоуровнево, но, учитывая, что группы можно вставлять, удалять или перемещать, расположение группы в программе записи может некоторое время отличаться от ее расположения в программе чтения (пока не будут применены изменения). Поэтому необходимо поддерживать дельту для отслеживания разницы. Эта дельта обновляется при вставках, удалениях и перемещениях и отражает "нереализованное расстояние, на которое должен переместиться писатель, чтобы соответствовать текущему слоту в читателе", как сказано в документации.
#### Применение изменений
Как мы уже много раз упоминали в этой главе, за это отвечает `Applier`. Текущий `Composer` делегирует этой абстракции полномочия по применению всех записанных изменений после композиции. Это то, что мы знаем как "материализация". Этот процесс выполняет список `Changes`, в результате чего обновляет таблицу слотов и интерпретирует хранящиеся в ней данные композиции, чтобы получить результат.

Время выполнения не зависит от того, как реализован `Applier`. Она опирается на публичный контракт, который должны реализовывать клиентские библиотеки. Это связано с тем, что Applier является `точкой` интеграции с платформой, поэтому он будет меняться в зависимости от сценария использования. Этот контракт выглядит следующим образом: 
```Kotlin
interface Applier<N> {
	val current: N
	fun onBeginChanges() {}
	fun onEndChanges() {}
	fun down(node: N)fun up()
	fun insertTopDown(index: Int, instance: N)
	fun insertBottomUp(index: Int, instance: N)
	fun remove(index: Int, count: Int)
	fun move(from: Int, to: Int, count: Int)
	fun clear()
}
```
Первое, что мы видим, - это параметр типа `N` в объявлении контракта. Это тип для узлов, которые мы применяем. Именно поэтому Compose может работать с общими графами вызовов или деревьями узлов. Он никогда не знает типа используемых узлов. `Applier` предоставляет операции для обхода дерева, вставки, удаления или перемещения узлов, но его не волнует тип этих узлов или то, как они в конечном итоге будут вставлены (это будет делегировано самим узлам).

Контракт также определяет, как удалить все дочерние узлы в заданном диапазоне из текущего узла или переместить дочерние узлы из текущего узла, чтобы изменить их положение. Операция `clear` определяет, как указать на корень и удалить все узлы из дерева, подготавливая `Applier` и его корень к использованию в качестве цели новой композиции в будущем.

`Applier` обходит все дерево, посещая и применяя все узлы. Дерево может быть пройдено сверху вниз или снизу вверх. Он всегда сохраняет ссылку на текущий узел, который он посещает и к которому применяет изменения. У него есть вызовы для начала и окончания применения изменений, которые `Composer` будет вызывать до и после, и он предоставляет средства для вставки сверху вниз или снизу вверх, а также для навигации сверху вниз (переход к дочернему узлу текущего узла) или снизу вверх (переход к родителю текущего узла).

Единственный ключевой момент здесь - выбирайте одну или другую стратегию для вставки, но никогда обе.
#### Как применяются изменения
Как мы уже описывали выше, клиентские библиотеки предоставляют реализации интерфейса `Applier`, одним из примеров является `UiApplier` для Android UI. Посмотрим на нем, что означает "применение узла" и как это приводит к появлению компонентов, которые мы можем видеть на экране в данном сценарии.
```Kotlin
internal class UiApplier(
	root: LayoutNode
) : AbstractApplier<LayoutNode>(root) {  

override fun insertTopDown(index: Int, instance: LayoutNode) {
// Ignored.
}

override fun insertBottomUp(index: Int, instance: LayoutNode) { 
	current.insertAt(index, instance)
}

override fun remove(index: Int, count: Int) { 
	current.removeAt(index, count)
}

override fun move(from: Int, to: Int, count: Int) { 
	current.move(from, to, count)
}

override fun onClear() { root.removeAll() }

override fun onEndChanges() {  
	super.onEndChanges()  
	(root.owner as? AndroidComposeView)?.clearInvalidObservations()
	} 
}
```
Первое, что мы видим, - это то, что тип `N` был определен как `LayoutNode`. Это тип узла, который Compose UI выбрал для представления узлов пользовательского интерфейса, которые будут отображаться.

Следующее, что мы замечаем - то, как он расширяет `AbstractApplier`. Это реализация по умолчанию, которая хранит посещенные узлы в стеке. Каждый раз, когда новый узел посещается вниз по дереву, он добавляет его в стек, а каждый раз, когда посетитель перемещается вверх, он удаляет последний посещенный узел с вершины стека. Как правило, эта функция является общей для всех приложений, поэтому, вероятно, было бы неплохо иметь ее в общем родительском классе.

Мы также видим, как `insertTopDown` игнорируется в `UiApplier`, поскольку в случае Android вставки будут выполняться снизу вверх. Как мы уже говорили выше, важно выбрать одну стратегию или другую, а не обе. В данном случае стратегия "снизу вверх" будет более подходящей, чтобы избежать дублирования уведомлений при вставке нового дочернего элемента. Эта разница в производительности была объяснена ранее.

Методы для вставки, удаления или перемещения узла делегируются самому узлу. `LayoutNode` - это то, как Compose UI моделирует узел UI, поэтому он знает все о родительском узле и его дочерних элементах. Вставка узла означает прикрепление его к новому родителю в заданной позиции (у него может быть несколько дочерних узлов). Перемещение узла - это, по сути, изменение порядка списка дочерних узлов для его родителя. Наконец, удаление узла означает просто удаление его из списка.

По окончании применения изменений можно вызвать функцию `onEndChanges()`, которая делегирует владельцу корневого узла последнее необходимое действие. Предполагается, что функция `onBeginChanges()` всегда вызывается до применения изменений, `поэтому onEndChanges()` нужно вызывать в конце. Это наблюдения моментальных снимков, предназначенные для автоматического повторного вызова компоновки или рисования при изменении значений, из которых они считываются и от которых зависят. Представьте, что узлы добавляются, вставляются, заменяются или перемещаются, и как это может повлиять на такие вещи, как измерение или компоновка
#### Прикрепление и рисование узлов
Как вставка узла в дерево (прикрепление его к родителю) означает, что мы в конечном итоге увидим его на экране? Ответ таков: узел знает, как нарисовать себя.
`LayoutNode` - это тип узла, выбранный для данного конкретного случая использования (Android UI). Когда реализация `UiApplier` делегирует ему вставку, все происходит в следующем порядке:
- Проверяем, что условия для вставки узла выполнены - например, у него еще нет родителя.
- Добавляем дочерний узел в список дочерних узлов, который он ведет, и обновляем список отсортированных дочерних узлов. Это параллельный список, поддерживаемый для быстрой сортировки в индексе Z.
- Присоедините новый узел к его родителю (владельцу). 

>Владелец находится в корне дерева и реализует **связь с базовой системой представлений**. Мы можем рассматривать его как тонкий интеграционный слой с Android. На самом деле он реализуется `AndroidComposeView`, который сам по себе является `View`. Владелец подключается к представлениям Android, и все операции по компоновке, рисованию, вводу и доступности осуществляются через них. Узел `LayoutNode` должен быть прикреплен к владельцу, чтобы отображаться на экране, и его владелец должен быть тем же владельцем, к которому прикреплен родитель. Владелец также является частью пользовательского интерфейса Compose.

![[layout node hierarchy.jpg]]
При присоединении узла выполняются некоторые дополнительные действия:
- Проверьте, что он еще не прикреплен или не пытается прикрепиться к владельцу, отличному от родительского. (Требование объяснено выше).
- Обновить семантику родителя, поскольку в семантическое дерево добавляется новый семантический узел. 
- Попросите владельца создать слой, который умеет рисовать содержимое `LayoutNode` с помощью Compose UI `Canvas`. Compose UI `Canvas` - это абстракция, которая реализована для Android путем обертывания родного Android `Canvas` и делегирования ему полномочий.
- Учитывая, что владелец реализован `AndroidComposeView` - реальным `View` - он предоставляет доступ ко всем примитивам `View`, которые используются для аннулирования после внесения изменений.
- Запросите повторную оценку для владельца и для родителя.
ПРОФИТ! Наконец-то мы знаем, как Compose UI материализует дерево узлов для Android. Реализация `Applier` делегирует это узлам, которые знают, как нарисовать себя на `Canvas` и выполнить необходимые аннулирования, чтобы изменения отразились на экране.
### Composition
В предыдущем разделе мы узнали много интересных подробностей о `Composer`. Мы знаем, как он записывает изменения для записи в таблицу слотов или чтения из нее, как эти изменения выдаются при выполнении CF во время композиции, и как эти записанные изменения применяются в конечном итоге. Но на самом деле мы ни слова не сказали (пока) о том, кто отвечает за создание `Composition`, как, когда это происходит и какие шаги при этом выполняются. 
Мы сказали, что у `Composer` есть ссылка на `Composition`, но это может заставить нас думать, что `Composition` создается и принадлежит `Composer`, в то время как на самом деле все наоборот. Когда композиция создается, она сама строит `Composer`. `Composer` становится доступным через механизм `currentComposer`, и он будет использоваться для создания и обновления дерева, управляемого `Composition`.
Точка входа клиентских библиотек в среду исполнения JC разделена на две части:
- Написание функций `Composable`: Это заставит их выдавать всю необходимую информацию и, следовательно, свяжет наш сценарий использования со средой выполнения.
- CF - это здорово, но они никогда не будут выполняться без композиции. Поэтому необходима еще одна точка входа: `setContent`. Это уровень интеграции с целевой платформой, и здесь создается и инициируется Composition.
#### Создание Composition
```Kotlin
internal fun ViewGroup.setContent( 
	parent: CompositionContext, 
	content: @Composable () -> Unit
): Composition {
    // ...
    val composeView = ...  
    return doSetContent(composeView, parent, content)
}

private fun doSetContent( 
    owner: AndroidComposeView, parent: CompositionContext,
    content: @Composable () -> Unit
): Composition {
	// ...  
	val original = Composition(UiApplier(owner.root), parent) // Here! 
	val wrapped = owner.view.getTag(R.id.wrapped_composition_tag) as? WrappedComposition ?: WrappedComposition(owner, original).also {
		owner.view.setTag(R.id.wrapped_composition_tag, it)
	} 
	wrapped.setContent(content) return wrapped
}
```
`WrappedComposition` - это декоратор, который знает, как связать `Composition` с `AndroidComposeView`, чтобы подключить его непосредственно к системе Android View. Он запускает управляемые эффекты для отслеживания таких вещей, как изменение видимости или доступности клавиатуры, и передает информацию об Android-контексте, который будет отображаться в `Composition` в виде `CompositionLocals`. (т. е. сам контекст, конфигурация, текущий владелец `LifecycleOwner`, текущий владелец `savedStateRegistry`, или представление владельца, среди прочего). Таким образом, все эти вещи становятся неявно доступными для всех наших CF.

Обратите внимание, как в `Composition` передается экземпляр `UiApplier`, который начинает указывать на корневой узел `LayoutNode` дерева. (`Applier` - это посетитель узлов, поэтому он начинает указывать на корневой узел). Это первый раз, когда мы явно видим, что именно клиентская библиотека отвечает за выбор реализации для `Applier`.

Также мы видим, как в конце вызывается `composition.setContent(content)`. `Composition#setContent` - это то, что устанавливает содержимое композиции. (Обновляет ее всей информацией, предоставленной `content`).

Независимо от варианта использования, при создании композиции может быть передан родительский `CompositionContext` (см. выше). Обратите внимание, что он может быть `null`. Родительский контекст (если он доступен) будет использоваться для логической связи новой композиции с существующей, так что инвалидации и `CompositionLocals` могут разрешаться между композициями, как если бы они были одной и той же.

При создании `Composition` также можно передать контекст `recompose`, который будет `CoroutineContext`, используемым `Applier` для применения изменений и, в конечном счете, материализации дерева. Если он не указан, то по умолчанию используется контекст, предоставленный рекомпозитором, а это `EmptyCoroutineContext`. Это означает, что Android, скорее всего, рекомпозирует на `AndroidUiDispatcher.Main`.

Точно так же, как создается композиция, она должна быть утилизирована - то есть: `composition.dispose()`, когда она больше не нужна. Именно тогда UI (или альтернативные варианты использования) для нее будут утилизированы. Мы можем сказать, что композиция привязана к своему владельцу. Иногда утилизация может быть немного скрыта, как в случае `ViewGroup.setContent` (за наблюдателем жизненного цикла), но оно всегда присутствует
#### Инициализация Composition
При создании новой композиции всегда следует вызов `composition.setContent(content)`. Фактически, именно здесь происходит первоначальное заполнение композиции (таблица слотов заполняется соответствующими данными).

Этот вызов делегируется родительскому `Composition` для запуска начального процесса `Composition` (напомним, что композиции и подкомпозиции связаны между собой через родительский `CompositionContext`):
```Kotlin
override fun setContent(content: @Composable () -> Unit) { 
	// ...
	this.composable = content  
	parent.composeInitial(this, composable)
}
```
Для субкомпозиций родителем будет другая композация. Для корневой родителем будет `Recomposer`. Но независимо от этого, логика выполнения начальной композиции в любом случае будет опираться на `Recomposer`, поскольку для субкомпозиций вызов `composeInitial` делегирует родительский вызов снова и снова, пока не достигнет корневой.

Поэтому вызов `parent.composeInitial(composition, content)` может быть переведен в `recomposer.composeInitial(composition, content),` и еще он выполняет несколько важных действий для наполнения начальной композиции:
- Делает снимок текущего значения всех объектов `State`. Эти значения будут изолированы от потенциальных изменений других снимков. Этот снимок **изменяемый**, но в то же время он потокобезопасный. Его можно безопасно модифицировать, не затрагивая другие существующие снимки состояния, поскольку любые изменения в любом из его объектов состояния будут происходить только для него, и он будет атомарно синхронизировать все эти изменения с глобальным общим состоянием на более позднем шаге.
- Значения `State` из этого снимка могут быть изменены только из блока, переданного при вызове `snapshot.enter(block: () -> T)`.
- При создании моментального снимка `Recomposer` также передает наблюдатели за любыми чтениями или записями в упомянутые объекты `State`, чтобы `Composition` могла получить соответствующее уведомление, когда они происходят. Это позволяет ей пометить затронутые области рекомпозиции как используемые, что заставит их перекомпоноваться, когда придет время.
- Входит в снимок - т.е.: `snapshot.enter(block)` - передавая следующий блок: `composition.composeContent(content)`. Именно здесь происходит собственно композиция. Действие входа позволяет `Recomposer` знать, что любые объекты `State`, прочитанные или записанные во время композиции, будут отслеживаться (сообщены `Composition`).
- Процесс композиции делегируется `Composer`. 
- После завершения композиции любые изменения объектов состояния вносятся только в текущий снимок состояния, поэтому настает время распространить эти изменения в глобальное состояние. Это происходит через `snapshot.apply()`.

А сейчас давайте подробно остановимся на самом процессе Composition, делегированном `Composer`. Вот как это происходит в общих чертах.
- Композиция не может быть запущена, если она уже запущена. В этом случае будет выброшено исключение, и новая композиция будет отброшена. Реентерабельная композиция не поддерживается.
- Если есть какие-либо ожидающие аннулирования, он копирует их в список аннулирования, который ведет `Composer` для `RecomposeScopes`, ожидающих аннулирования.
- Перемещает флаг `isComposing` в true, так как композиция вот-вот начнется.
- Вызывает `startRoot()` для запуска композиции, которая запустит корневую группу для композиции в таблице слотов и инициализирует другие необходимые поля и структуры.
- Вызывает `startGroup`, чтобы запустить группу для содержимого в таблице слотов.
- Вызывает лямбду `content`, чтобы она передала все свои изменения.
- Вызывает `endGroup`, чтобы завершить группу в таблице слотов.
- Вызывает `endRoot()`, чтобы завершить композицию.
- Перемещает флаг `isComposing` в `false`, так как композиция завершена.
- Очищает другие структуры, хранящие временные данные.
#### Применение изменений после начальной композиции
После первоначальной композиции `Applier` получает уведомление о необходимости применить все изменения, записанные в процессе: `composition.applyChanges()`. Это делается через `Composition`, который также вызывает `applier.onBeginChanges()`, проходит по списку изменений, выполняя их все и передавая каждому изменению необходимые экземпляры `Applier` и `SlotWriter`. Наконец, после того как все изменения применены, он вызывает `applier.onEndChanges()`. Это естественный процесс.

После этого происходит диспетчеризация всех зарегистрированных `RememberedObservers`, так что любые классы, реализующие контракт `RememberObserver`, могут быть уведомлены о входе или выходе из `Composition`. Такие вещи, как `LaunchedEffect` или `DisposableEffect`, реализуют его, чтобы ограничить эффект жизненным циклом `Composable` в пределах `Composition`.

Сразу после этого все побочные эффекты запускаются в том же порядке, в котором они были записаны.
#### Дополнительная информация о композиции
Композиция знает о своих ожидающих инвалидациях для перекомпозиции. Она также знает, составляет ли она композицию в данный момент. Это знание может быть использовано для мгновенных инвалидаций (если она есть) или их откладывания. Оно также может быть использовано `Recomposer` для отбрасывания рекомпозиций.

Среда выполнения опирается на вариант композиции под названием `ControlledComposition`, который добавляет несколько дополнительных функций, чтобы им можно было управлять извне. Таким образом, `Recomposer` может организовывать инвалидацию и дальнейшую рекомпозицию. Хорошими примерами этого являются такие функции, как `composeContent` или `recompose`. `Recomposer` может вызывать эти действия в композиции, когда это необходимо.

Композиция предоставляет средства для обнаружения того, что набор объектов наблюдается сам по себе, чтобы принудительно перекомпоновать его, когда они меняются. Например, это используется `Recomposer` для принудительной рекомпозиции в дочерней композиции, когда `CompositionLocal` изменяется в родительской композиции. Помните, что для этого композиции связаны через родительский `CompositionContext`.

Иногда во время композиции обнаруживается ошибка, в этом случае она может быть прервана, что практически равносильно сбросу `Composer` и всех его ссылок/стеков и всего остального.

`Composer` считает, что он пропускает рекомпозицию, если он не вставляет и не использует повторно, нет недействительных провайдеров (поскольку это потребовало бы рекомпозиции) и текущий `RecomposeScope` не требует рекомпозиции. 
### The Recomposer
Как он создается и когда начинает работать? Как он начинает прослушивать инвалидации для автоматического запуска рекомпозиции?

`Recomposer` управляет `ControlledComposition` и запускает рекомпозиции, когда это необходимо, чтобы в конечном итоге применить к нему обновления. Он также определяет, в каком потоке выполнять композицию или рекомпозицию, и какой поток использовать для применения изменений.
#### Создание Recomposer
Точкой входа в JC для клиентских библиотек является создание композиции и вызов `setContent` над ней - см. [[Среда выполнения Compose#Создание Composition|Создание Composition]]. При создании `Composition` необходимо указать ее родителя. Учитывая, что родителем корневой композиции является `Recomposer`, это также момент для его создания.

Эта точка входа является связующим звеном между платформой и средой выполнения Compose и представляет собой код, предоставляемый клиентом. В случае Android это Compose UI. Эта библиотека создает `Composition` (который внутренне создает свой собственный `Composer`) и `Recomposer` для использования в качестве родителя.

>Обратите внимание, что каждый потенциальный вариант использования для каждой платформы, как мы уже выяснили, склонен создать свой собственный Composition, и точно так же, скорее всего, он создаст свой собственный Recomposer.

Когда мы хотим использовать Compose на Android `ViewGroups`, мы вызываем `ViewGroup.setContent`, который в конечном итоге, после некоторых перенаправлений, делегирует создание родительского контекста фабрике `Recomposer`:
```Kotlin
fun interface WindowRecomposerFactory {

	fun createRecomposer(windowRootView: View): Recomposer

	companion object {  
		val LifecycleAware: WindowRecomposerFactory = WindowRecomposerFactory {
			rootView -> rootView.createLifecycleAwareViewTreeRecomposer()
		}
	}
}
```

Эта фабрика создает `Recomposer` для текущего окна. Передача ссылки на корневое представление необходима для вызова `createRecomposer`, поскольку созданный `Recomposer` будет поддерживать жизненный цикл, то есть будет связан с `ViewTreeLifecycleOwner` в корне иерархии представлений. Это позволит отменить (выключить) `Recomposer`, когда дерево представлений, например, не прикреплено, что важно для предотвращения утечки процесса рекомпозиции. Этот процесс моделируется как suspend-функция, которая в противном случае будет утекать.

>Инфикс, обозначающий, что это такое, приведен ниже: В Compose UI все происходящее в UI координируется с помощью `AndroidUiDispatcher`, который по этой причине связан с экземпляром `Choreographer` и обработчиком для главного `Looper`. Этот диспетчер выполняет диспетчеризацию событий во время обратного вызова обработчика или на этапе кадра анимации `Choreographer`, в **зависимости от того, что наступит раньше**. С ним также связан `MonotonicFrameClock`, который использует `suspend` для координации рендеринга кадров. Это то, что управляет всем UX в Compose, и такие вещи, как анимация, во многом зависят от этого для достижения плавного опыта в синхронизации с системными кадрами.

Первое, что делает фабричная функция, - создает `PausableMonotonicFrameClock`. Это обертка над монотонными часами `AndroidUiDispatcher`, которая добавляет поддержку ручной приостановки отправки событий `withFrameNanos` до их возобновления. Это делает его полезным для случаев, когда кадры не должны создаваться в течение определенных периодов времени, например, когда окно, в котором находится UI, больше не видно. Любой `MonotonicFrameClock` также является `CoroutineContext.Element`, что означает, что его можно комбинировать с другими `CoroutineContexts`. При инстанцировании `Recomposer` мы должны предоставить ему `CoroutineContext`. Этот контекст создается с помощью комбинации контекста текущего потока из `AndroidUiDispatcher` и только что созданных часов с паузой.
```Kotlin
val contextWithClock = currentThreadContext + (pausableClock ?: EmptyCoroutineContext)
val recomposer = Recomposer(effectCoroutineContext = contextWithClock)
```
Этот комбинированный контекст будет использоваться `Recomposer` для создания внутренней `Job`, чтобы гарантировать, что все эффекты композиции или рекомпозиции могут быть отменены при выключении `Recomposer`. Это будет необходимо, например, когда окно Android уничтожается или отсоединяется. **Этот контекст будет использоваться для применения изменений** после композиции/рекомпозиции, а также будет контекстом по умолчанию, используемым `LaunchedEffect` для запуска эффектов. Таким образом, эффекты будут запускаться в том же потоке, который мы используем для применения изменений, а в Android это обычно главный поток. Конечно, мы всегда можем по своему усмотрению спрыгнуть с основного потока в рамках наших эффектов.

Наконец, создается coroutine scope с использованием того же объединенного контекста: например: `val runRecomposeScope = CoroutineScope(contextWithClock)`. Эта область будет использоваться для запуска `Job` рекомпозиции (suspend-функция), которое будет ожидать инвалидации и соответственно запускать рекомпозицию. Давайте заглянем в код и обсудим некоторые идеи по этому поводу.
```Kotlin
viewTreeLifecycleOwner.lifecycle.addObserver(
	object : LifecycleEventObserver {
		override fun onStateChanged(lifecycleOwner: LifecycleOwner, event: Lifecycle.Event) {
			val self = this
			when (event) { Lifecycle.Event.ON_CREATE ->
				runRecomposeScope.launch(start = CoroutineStart.UNDISPATCHED) { 
					try {
						recomposer.runRecomposeAndApplyChanges() 
					} finally {
			            // After completion or cancellation
			            lifecycleOwner.lifecycle.removeObserver(self)
					}
        }
		        Lifecycle.Event.ON_START -> pausableClock?.resume()
		        Lifecycle.Event.ON_STOP -> pausableClock?.pause()
		        Lifecycle.Event.ON_DESTROY -> {
		          recomposer.cancel()
		        }
			} 
		}
	} 
)```
Вот здесь-то все и происходит. К жизненному циклу дерева представлений прикреплен наблюдатель, который будет использовать часы с паузами для возобновления и приостановки отправки событий при запуске и остановке дерева представлений соответственно. Он также выключит (отменит) `Recomposer` при уничтожении и запустит задание рекомпозиции при создании.

Задание рекомпозиции запускается функцией `recomposer.runRecomposeAndApplyChanges()`, которая является функцией приостановки, упомянутой выше, которая будет ожидать аннулирования всех связанных композиций (и их `RecomposeScopes`), рекомпозировать их и в конечном итоге применять новые изменения к связанной с ними композиции.

Эта фабрика - то, как Compose UI порождает `Recomposer`, связанный с жизненным циклом Android. Она хорошо работает в качестве примера того, как `Recomposer` создается в точке интеграции с платформой, вместе с `Composition`. В качестве иллюстрации здесь мы снова видим, как создается композиция при настройке содержимого для `ViewGroups`:
```Kotlin
internal fun ViewGroup.setContent(  
	parent: CompositionContext, // Recomposer is passed here! 
	content: @Composable () -> Unit
): Composition {
	// ...
	val composeView = ...  
	return doSetContent(composeView, parent, content)
}

private fun doSetContent(  
	owner: AndroidComposeView, 
	parent: CompositionContext, 
	content: @Composable () -> Unit
): Composition {
// ...  
	val original = Composition(UiApplier(owner.root), parent) // Here! 
	val wrapped = owner.view.getTag(R.id.wrapped_composition_tag)
		as? WrappedComposition ?: WrappedComposition(owner, original).also {
			owner.view.setTag(R.id.wrapped_composition_tag, it)
		} 
	wrapped.setContent(content) 
	return wrapped
}
```
В качестве родителя будет выступать `Recomposer`, который будет предоставлен вызывающей стороной `setContent`, а в данном случае это `AbstractComposeView`.
### Процесс рекомпозиции
Функция `recomposer.runRecomposeAndApplyChanges()` вызывается, чтобы начать ожидать инвалидаций и автоматически рекомпозироваться, когда они происходят. 
При вызове `recomposer.runRecomposeAndApplyChanges()` первым делом регистрируется наблюдатель для распространения изменений. Когда это происходит, этот наблюдатель просыпается и добавляет все эти изменения в список инвалидаций снимка, который распространяется среди всех известных композиторов, чтобы они могли записать, какие части композиции нужно перекомпоновать. Проще говоря, этот наблюдатель - ступенька для запуска автоматической перекомпоновки при изменении состояния.

После регистрации наблюдателя использования снимка `Recomposer` аннулирует все композиции, чтобы принять за отправную точку момент, когда все изменилось. - Любые изменения, произошедшие до этого момента, не были отслежены. - Затем он приостанавливается до тех пор, пока не появится работа, доступная для перекомпоновки. "Наличие доступной работы" означает наличие любых ожидающих аннулирования снимков состояния или аннулирования композиций, поступающих от `RecomposeScopes`.

Следующее, что делает `Recomposer`, - это использует монотонные часы, предоставленные при создании, и вызывает `parentFrameClock.withFrameNanos {}` для ожидания следующего кадра. Вся остальная работа будет выполняться в это время, а не раньше. Цель этого - объединить изменения в кадре. Внутри этого блока `Recomposer` сначала диспетчеризует кадры монотонного счетчика времени на предмет потенциальных ожидающих (например, анимации). В результате могут возникнуть новые инвалидации, которые также необходимо отследить (например, переключение условной `Composable` при завершении анимации).

А теперь настало время для настоящего действия. `Recomposer` берет все ожидающие аннулирования снимки, или, другими словами, все значения `State`, измененные с момента последнего вызова рекомпозиции, и записывает все эти изменения в `Composer` как ожидающие рекомпозиции. Также могут быть недействительные композиции - через `composition.invalidate()`, - например, когда состояние записывается в лямбду `Composable`. Для каждой из них `Recomposer` выполняет рекомпозицию (раздел об этом ниже) и добавляет ее в список композиций с изменениями, ожидающими применения.

Рекомпозиция означает пересчет всех `Changes`, необходимых для состояния композиции (таблица слотов) и материализованного дерева (`Applier`), как мы уже выяснили. [[Среда выполнения Compose#Инициализация Composition|Мы уже видели, как это делается]]. Позже он находит потенциальные "хвостовые" рекомпозиции, которые должны быть составлены из-за изменения значения в композиции, и также диспетчеризует их для рекомпозиции. Это может произойти, например, если `CompositionLocal` изменился в родительской композиции и был прочитан в дочерней композиции, которая в остальном была действительна. Наконец, `Composer` перебирает все композиции с изменениями, которые нужно применить, и вызывает на них` composition.applyChanges()`. После этого он обновляет состояние.
#### Конкурентная рекомпозиция
`Recomposer` имеет возможность выполнять рекомпозиции конкурентно, даже если Compose UI не использует эту возможность. Однако любые другие клиентские библиотеки могут использовать ее в зависимости от своих потребностей. `Recomposer` предоставляет параллельный аналог функции `runRecomposeAndApplyChanges`, который называется `runRecomposeConcurrentlyAndApplyChanges`. Это еще одна функция приостановки для ожидания аннулирования снимков состояния и запуска автоматической рекомпозиции, как и первая, но с той лишь разницей, что последняя будет выполнять рекомпозицию аннулированных Композиций в `CoroutineContext`, предоставленном извне.
```Kotlin
suspend fun runRecomposeConcurrentlyAndApplyChanges(
    recomposeCoroutineContext: CoroutineContext
) { /*...*/ }
```
Эта функция приостановки создает свой собственный `CoroutineScope`, используя переданный контекст, и использует его для порождения и координации всех дочерних заданий, созданных для всех необходимых параллельных рекомпозиций.
#### Состояния Recomposer
В течение своей жизни `Recomposer` переключается в ряд состояний:
```Kotlin
enum class State {
	ShutDown,
	ShuttingDown,
	Inactive,
	InactivePendingWork,
	Idle,
	PendingWork
}
```
- `ShutDown`: `Recomposer` был отменен и работа по очистке завершена. Больше не может быть использован.
- `ShuttingDown`: `Recomposer` был отменен, но он все еще находится в середине процесса очистки. Больше не может быть использован.
- `Неактивен`: `Recomposer` будет игнорировать аннулирования от `Composer`-ов и не будет запускать перекомпоновку. Для начала прослушивания необходимо вызвать `runRecomposeAndApplyChanges`. Это начальное состояние `Recomposer` после создания.
- `InactivePendingWork`: Существует вероятность того, что `Recomposer` неактивен, но уже имеет некоторые отложенные эффекты, ожидающие кадра. Кадр будет создан, как только `Recomposer` начнет работать.
- `Idle`: `Recomposer` отслеживает композицию и инвалидации снимков, но в настоящее время нет никакой работы.
- `PendingWork`: `Recomposer` был уведомлен о предстоящей работе и уже выполняет ее или ожидает возможности ее выполнить. (Мы уже описали, что означает "ожидание работы" для `Recomposer`).
