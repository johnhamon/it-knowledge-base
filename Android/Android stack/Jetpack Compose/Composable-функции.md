[Немного про то, во что компилятся `Composable`-функции](https://medium.com/androiddevelopers/under-the-hood-of-jetpack-compose-part-2-of-2-37b2c20c6cdd)
Дерево CF компилируется в буферное окно - массив, позволяющий вставку, удаление и нахождение элемента за константное время ценой большого размера структуры.

`@Composable` меняет тип функции, к которой применяется, и накладывает некоторые ограничения и привносит ей некоторые свойства, с которыми потом работает JC. Среда выполнения Compose ожидает этих свойств у функций, чтобы предполагать некоторое их поведение и применить оптимизации среды выполнения.

Любая функция, аннотированная как `@Composable`, преобразуется компилятором JC в функцию, которая неявно получает в качестве параметра экземпляр контекста `Composer`, а также передает этот экземпляр своим дочерним `Composable`. Мы можем рассматривать это как инжектируемый неявный параметр, к которому время выполнения и разработчик не имеют никакого отношения.
```kotlin
@Composable
fun Counter() {
	var count by remember { mutableStateOf(0) }
	Button(
		text="Count: $count",
		onPress={ count += 1 }
	)
}
```
```Kotlin
fun Counter($composer: Composer) {
	$composer.start(123)
	var count by remember { mutableStateOf(0) }
	Button(
		text="Count: $count",
		onPress={ count += 1 }
	)
	$composer.end()
}
```
Это позволяет перенаправить контекст `Composer` вниз по дереву, так что он всегда будет доступен на любом уровне, если дерево состоит только из `Composable`-функций. Компилятор убедится в этом, наложив очень строгое правило: `Composable`-функции могут быть вызваны только из других `Composable`-функций. Другими словами, это будет требуемый контекст вызова.

Налагая это требование, JC может гарантировать, что необходимая для выполнения информация всегда будет доступна любому поддереву. Composable будет использовать инжектированный экземпляр `Composer` для эмиссии этих изменений во время композиции, а последующие рекомпозиции будут зависеть от изменений, эмитированных предыдущими выполнениями функции. Это связь между производственным кодом, который мы пишем как разработчики, и средой выполнения `Compose`. Эта связь дает нам возможность информировать среду выполнения о форме дерева, чтобы она могла построить свое представление в памяти и выполнить свою магию.
#### Рекомпозиция
Рекмопозиция CF означает ее повторное выполнение. Все CF могут перезапускаться рекомпозицией, но только когда это необходимо. `@NonRestartableComposable` отключает перезапускаемость. 

- `Composable`-функции идемпотентны. Рекомпозиция может случаться из-за разных причин, и функция может быть пересобрана много раз, поэтому важно, чтобы она всегда возвращала одно и то же. Рекомпозиция идет по дереву функций сверху вниз проверяя, какие узлы нуждаются в рекомпозиции. Когда мы говорим об "умной рекомпозиции", мы имеем в виду, что части дерева, которые не зависят от изменяемых данных, могут оставаться неизменными, поэтому они не рекомпозируются. Это дает значительный прирост производительности.
- `Composable`-функции чистые. Поэтому могут выполняться в любом порядке, в т.ч. параллельно. Поэтому бизнес-логика, вызывающая побочные эффекты, должна находиться в другом слое. Также побочные эффекты могут неконтролируемо вызывать рекомпозицию.
- Рекомпозиция сама является побочным эффектом, но он приемлем, потому что дерево функций нужно для рисования UI, а не само по себе
- Предполагается, что CF будут быстрыми, поскольку их можно вызывать многократно. Именно поэтому они предназначены для того, чтобы производить изменения узлов в дереве `Composable`, а не материализовывать реальный UI сразу же. Например, функция `Composable` может вызываться для каждого кадра анимации. Любые вычисления, требующие больших затрат, следует перегружать в корутины и всегда оборачивать в один из обработчиков эффектов с учетом жизненного цикла. Если побочные эффекты учитывают жизненный цикл и приостанавливаются, компилятор JC может гарантировать, что мы используем их правильно и в соответствии с тем, что ожидает среда выполнения.
#### Позиционная мемоизация
CF способны к мемоизации - кэшировать свой результат на основе входных данных, чтобы каждый раз не вычислять. Позиционная мемоизация основана на этой идее, но с ключевым отличием. CF постоянно знают о своем местоположении в дереве. Среда выполнения будет различать вызовы одной и той же CF, предоставляя им идентификатор, уникальный в пределах родителя. Этот идентификатор генерируется, помимо прочего, на основе позиции вызова CF. Если CF вызывается всегда из одной и той же позиции, то CR положится на порядок вызовов, чтобы присвоить CF id. Если этот порядок изменится (что-то вставили в середину, например), то придется рекомпозировать. Чтобы избежать рекомпозиций, *можно присвоить идентификаторы элементам самостоятельно*.

Когда мы говорим, что CF производит узел в `Composition`, информация относящаяся к вызову, сохраняется в `Composer`
#### remember
Умеет читать из таблицы слотов, чтобы получить свой результат. Когда функция вызывается, она ищет вызов функции в таблице и возвращает кэшированный результат, если он доступен. В противном случае она вычислит его и сохранит  перед возвратом, чтобы его можно было извлечь позже.

В JС мемоизация не является традиционной "общеприкладной" мемоизацией. `remember` использует позиционную мемоизацию, чтобы получить кэшированное значение из контекста, ограниченного вызывающим его `Composable`. Это означает, что он обращается к таблице и ищет значение в диапазоне слотов, в которых, как ожидается, будет храниться информация для этого `Composable`. Это делает его более похожим на синглтон в этой области, поскольку он будет вычислять значение только во время начальной композиции, но при каждой повторной композиции он будет получать кэшированное значение. Но если тот же `Composable` будет использован в другой композиции, или тот же вызов функции будет запомнен из другой композиции, запомненное значение будет другим экземпляром.
